TypeAhead & Autocomplete System Design

A scalable, low-latency suggestion engine serving billions of queries daily

Overview

A TypeAhead/Autocomplete system powers real-time search suggestions as users type, reducing query time and improving UX. This document covers designing a system that handles 5 billion queries per day (58K QPS peak) with <100ms end-to-end latency at p99, using a Trie as the core data structure, Redis for caching prefixes, and Cassandra for persistent trie snapshots.

You'd encounter this in a system design interview at top-tier tech companies (Google, Amazon, Meta, Microsoft) as a real product problem with clear scale challenges and elegant distributed systems trade-offs.

Functional Requirements

  1. Autocomplete API: Given a prefix, return top-10 ranked suggestions within 100ms
  2. Ranking: Suggestions ranked by relevance (popularity + recency + user context)
  3. Freshness: Daily updates to rankings; weekly updates to vocabulary (new terms)
  4. Scale: Handle 5B queries/day globally; peak 58K QPS (7x average)
  5. Correctness: Exact-match filtering; support prefix matching for UTF-8 strings
  6. Spelling: Fuzzy matching support for typos (e.g., "appl" → "apple")
  7. Analytics: Log queries for trending, seasonal patterns, and model training
  8. Monitoring: Alert on latency SLA breach, cache hit rate drop, or trie staleness

High-Level Architecture

Component Breakdown

Client Layer: Web/mobile apps that debounce user input (100-200ms) before sending requests to reduce QPS. XHR or WebSocket for real-time suggestions.

API Gateway: Load balances requests across suggestion servers. Implements rate limiting (e.g., 1000 req/sec per user), request validation, and routing. Talks to DNS for service discovery.

Suggestion Service (Stateless): Core logic tier. Receives prefix, checks Redis cache first (90% hit rate), falls back to Cassandra trie lookup, ranks results, and returns JSON. Horizontally scalable; 10-20 instances per region. Caches recently built Trie objects in process memory for faster traversals.

Redis Cache: Stores hot 1-3 character prefixes → JSON array of top-10 suggestions + scores. TTL=1 hour (auto-refresh before expiry). Cluster mode for high availability; ~3-4 GB per region.

Cassandra Cluster: Persistent Trie snapshots. Each node holds range of terms (e.g., "a"-"az", "ba"-"bz"). Schema: key = trie_version, value = serialized_trie_binary. Weak consistency model acceptable (eventual consistency fine for stale suggestions). 3x replication across availability zones.

Kafka Queue: Ingests query events (prefix, user_id, timestamp, result_size). Stream processors aggregate counts by prefix hourly. Enables trending analysis, model training, and behavior analytics.

Stream Processor & Batch Jobs: Daily: recompute top-10 suggestions per prefix using popularity + recency decay. Weekly: scan Kafka logs for new terms, rebuild Trie, blue-green deploy to Cassandra. MapReduce on Hadoop cluster; ~30 min per run.

Core Data Structure: Trie

A Trie (prefix tree) is the foundational data structure. Each node represents a character; paths encode strings. We augment each node with top-10 suggestions and scores to avoid full tree traversal.

TrieNode Class (Python)

class TrieNode: def __init__(self): self.children = {} # char → TrieNode self.is_end_of_word = False self.frequency = 0 # Global frequency of this term self.top_suggestions = [] # List[Tuple[str, score]] def insert(self, word: str, freq: int): """Insert word into trie; update top_suggestions at each node.""" node = self for char in word: if char not in node.children: node.children[char] = TrieNode() node = node.children[char] # Update top-10 at this node node.top_suggestions.append((word, freq)) node.top_suggestions.sort(key=lambda x: -x[1]) node.top_suggestions = node.top_suggestions[:10] node.is_end_of_word = True node.frequency = freq def search(self, prefix: str) -> List[Tuple[str, int]]: """Search prefix; return top-10 suggestions.""" node = self for char in prefix: if char not in node.children: return [] node = node.children[char] return node.top_suggestions[:10]

Complexity Analysis

Operation Time Complexity Space Complexity Notes
Insert word O(m) where m = word length O(m) Traverse m characters; update top-10 at each level
Search prefix O(m) + O(1) O(1) Traverse m chars; return cached top-10 at final node
Delete word O(m) O(1) Mark as not end-of-word; clean up empty branches
Autocomplete O(m + 1) O(1) Search + return cached list; super fast!

Memory Layout

Per TrieNode: - children dict (HashMap): ~48 bytes (Python object overhead) + 8 bytes per child entry × ~26 = ~256 bytes - is_end_of_word: 1 byte - frequency: 8 bytes (long) - top_suggestions: 10 × (string ref + score) = 10 × 16 bytes = 160 bytes Total per node: ~425 bytes For 5M unique terms, avg word length 8: - Total nodes: ~5M × 8 = 40M nodes - Total memory: 40M × 425 bytes ≈ 17 GB (uncompressed in memory) - Serialized (gzip): ~2-3 GB in Cassandra

Key optimization: Pre-compute and cache top-10 suggestions at every node during trie rebuild. This turns a search query into a single O(m) traversal + O(1) array return, eliminating expensive graph traversals.

Ranking & Scoring Algorithm

Scoring Formula

score = (popularity_score × 0.6) + (recency_score × 0.3) + (personalization_score × 0.1) where: popularity_score = log₁₀(1 + query_count_past_week) [0-5 range] recency_score = 1 / (1 + days_since_peak × decay_factor) [0-1] decay_factor = 0.5 (halves score every 14 days) personalization_score = [0,1] if user has search history, else 0

Recency Decay Function

def recency_score(days_since_peak: float, decay_factor: float = 0.5) -> float: """Exponential decay for trending terms.""" return decay_factor ** (days_since_peak / 14) # Example: Term peaked 14 days ago → score = 0.5 # Term peaked 28 days ago → score = 0.25 # New term (0 days) → score = 1.0

Algorithm Comparison

Approach Pros Cons Recommendation
Pure Frequency
(count only)		 Simple, fast Stale results; past events dominate ❌ Avoid
Time-Decay
(exponential)		 Captures trends; current events boosted Ignores all-time popularity ⚠️ Use as blend
BM25 Variant
(TF-IDF style)		 Query-aware scoring; statistical sound Expensive to compute at scale; slower ranking ⚠️ For offline ranking
ML Ranking
(neural network)		 Captures complex signals; state-of-art High latency; model serving overhead ⭐ Future work; not for MVP
Weighted Blend
(popularity + recency)		 Balances stability + trends; fast; interpretable Requires tuning weights ✅ Recommended
Key Insight: Pre-compute scores offline during batch trie rebuild, not at query time. Bake scores into top-10 suggestions at each prefix node. This makes queries O(m) + O(1) return, avoiding real-time ML inference.

Database & Persistence

Why Cassandra for Trie Storage?

  • Write-optimized: Weekly trie rebuilds write 100s of GB efficiently; LSM tree structure
  • Distributed: Scales horizontally; 3x replication across AZs for durability
  • Eventual consistency: Acceptable for this use case; stale suggestions are okay
  • TTL support: Auto-delete old trie versions; keeps disk usage bounded
  • Range queries: Scan trie snapshots by version; support blue-green deploys
  • Operational maturity: Battle-tested at scale (Netflix, Uber, Airbnb)

Cassandra Schema

CREATE TABLE typeahead.trie_snapshots ( version TEXT, -- e.g., "v202403-en" language_code TEXT, -- e.g., "en", "es", "fr" partition_key INT, -- 0-99 for sharding across nodes trie_binary BLOB, -- Gzipped serialized trie created_at TIMESTAMP, PRIMARY KEY (version, language_code, partition_key) ) WITH CLUSTERING ORDER BY (language_code ASC, partition_key ASC) AND default_time_to_live = 604800; -- 7 days

Serialization Format

Binary Trie Format: [Header: 8 bytes] - magic: "TRIE" (4 bytes) - version: 1 (2 bytes) - term_count: uint32 (4 bytes) [Nodes: variable] - For each node in DFS pre-order: [char: 1 byte] [is_end: 1 bit] [score: float32: 4 bytes] [child_count: varint: 1-3 bytes] [top_suggestions: 10 × (string_offset, score)] [String Pool: variable] - Deduplicated terms; offsets in nodes point here Compression: gzip reduces 500GB → 50-100GB Serialization: 30 seconds (Python pickle) or 5 seconds (Protobuf/FlatBuffers)

Snapshot Strategy

  1. Weekly Full Snapshot: Sunday 2 AM UTC. Rebuild entire trie from logs. File size ~100 GB (gzipped). Deploy to staging Cassandra cluster first.
  2. Blue-Green Deployment: Keep two active Cassandra versions (e.g., v202403-en, v202410-en). Write to green; once verified, flip traffic. If issues, rollback to blue.
  3. TTL Auto-Cleanup: Set TTL=7 days. Old snapshots auto-delete; bound disk usage at ~500 GB per language.
  4. Incremental Updates: Daily ranking updates (popularity scores) stored separately in Redis (cheaper than Cassandra writes). Baked into next weekly trie rebuild.
Tip: Version trie snapshots by date + language (v202403-en, v202403-es). Store both in Cassandra under same table; use version + language_code as composite key for fast lookups.

Caching Layer

Redis Data Model

Key: "{language}:{prefix}" [e.g., "en:ap"] Value: JSON array of suggestions [ {"term": "apple", "score": 4.82}, {"term": "app", "score": 4.71}, ... ] Size per entry: ~500-800 bytes Hot prefixes (1-3 chars): ~3,500 keys × 800 bytes = 2.8 GB per language Hot prefixes (4 chars): ~17K keys × 1.2 KB = 20.4 GB (too much; skip caching)

Cache Population Strategy

Trigger Timing Implementation
Pre-warm on trie deploy Before blue-green flip Batch job: iterate 1-3 char prefixes, pre-compute suggestions, bulk insert to Redis
Lazy cache-miss On every query miss Suggestion service: compute from Cassandra trie, write to Redis with TTL
Refresh before expiry Every hour (TTL=1 hr) Background job: poll hot prefixes, check TTL, refresh if < 15 min left

TTL Strategy

TTL = 1 hour (3600 seconds) Reasoning: - Popularity scores updated daily; 1-hour staleness acceptable - Popular prefixes refreshed in background before expiry (avoid thundering herd) - Unpopular 4+ char prefixes not cached; recomputed on-demand - Warm prefixes (e.g., "a", "the") always cached via refresh job

Hit Rate Optimization

Tactic Expected Hit Rate Lift Implementation
Aggressive 1-char prefix caching +40% 26 entries; always keep in Redis; refresh every 30 min
Top-1000 queries pre-cached +30% Global trending; update every 6 hours from logs
User-specific overlays (personalization) +15% Cache per-user prefix→suggestions; 5-min TTL; size limit 100 MB
Predictive prefetch +5% On prefix "ap" query, pre-fetch "app", "apple" in background

Target hit rate: >90%. At 100K QPS, 10% cache miss = 10K cache misses/sec. Cassandra trie lookups at 10K/sec is manageable but risks overload. Invest in warming.

Warning: Cache stampede risk. If prefix expires mid-peak traffic (e.g., "vote" during election day), all 10K/sec requests hit Cassandra simultaneously. Mitigate with: (1) background refresh job, (2) probabilistic early expiry + refresh, (3) cache lock pattern.

Trie Update Pipeline

Pipeline Timeline

Frequency Task Duration Output
Real-time Stream queries to Kafka topic (async) < 1 sec Query event logged; available in 7-day window
Hourly Stream processor: aggregate hourly counts by prefix ~10 min Counters in Kafka topics; used for trending analysis
Daily (2 AM UTC) Recompute top-10 ranking per prefix; update Redis ~15 min New scores in Redis; used in next trie rebuild
Weekly (Sunday 2 AM) MapReduce: rebuild entire trie (new vocab + scores) ~30 min New trie version uploaded to staging Cassandra
Post-rebuild Validation: unit tests, canary traffic (1%), P99 latency check ~30 min Pass/fail; if pass, flip to production

MapReduce Pseudocode

# Hadoop MapReduce job for weekly trie rebuild def mapper(key, line): # Input: Kafka log entry {prefix, timestamp, user_id, ...} data = parse_json(line) prefix = normalize(data['prefix']) timestamp = data['timestamp'] yield prefix, (1, timestamp) def reducer(prefix, values): # Aggregate: count occurrences, compute score count = sum(v[0] for v in values) latest_ts = max(v[1] for v in values) if count > 100: # Frequency threshold score = compute_score(count, latest_ts) yield prefix, (term, score) # Post-reducer: Sort by score, build trie def build_trie(sorted_terms): trie = TrieNode() for term, score in sorted_terms: trie.insert(term, score) return gzip_serialize(trie)
Tip: Run a canary deployment: deploy new trie to 1% of suggestion service instances. Compare latency, cache hit rate, error rate vs. old trie for 1 hour. If metrics match, flip 100%.

Request Path Deep Dive

Request Flow (Step-by-step)

  1. User Input: User types "ap" in search box. Browser debounces for 100ms to batch rapid key presses.
  2. XHR/Fetch: After debounce, client sends GET /api/suggestions?prefix=ap&lang=en&limit=10
  3. API Gateway: Request hits load balancer; routed to least-loaded suggestion service instance (5ms network).
  4. Service: Redis Cache:
    • HIT (90%): Service queries Redis for key en:ap. Gets 10 pre-ranked suggestions in 5ms. Returns JSON immediately.
    • MISS (10%): Key not in cache. Fall through to Cassandra.
  5. Service: Cassandra Trie Lookup (cache miss only):
    • Service queries Cassandra for current trie version snapshot.
    • Trie is memory-mapped; traverse "a" → "p" nodes (2 hops, O(m)).
    • Fetch top-10 suggestions from final node (pre-computed at trie build time).
    • Time: 30-40ms (Cassandra latency + deserialization).
  6. Cache Warmth (miss path): Write result back to Redis with TTL=1 hour. Future queries for "ap" hit cache.
  7. Response Serialization: Convert suggestions to JSON. ~1.5 KB response.
  8. Network to Client: Send response (5-15ms latency).
  9. Client Render: Browser renders dropdown. User sees suggestions ~120-150ms after typing.
Key Insight: The latency budget is dominated by network round-trips. To hit p99 < 100ms, you need: (1) aggressive caching (≥90% hit rate), (2) low-latency cache (Redis in-memory), (3) regional deployment (CDN edge caches in user regions). Computation is negligible.

Partitioning Strategy

Challenge: Hotspot Distribution

Queries follow a Zipfian distribution: "the", "a", "and" dominate. English has ~3,500 hot 1-3 char prefixes, but their popularity is extremely skewed. "a" gets 10x more queries than "z".

Approach 1: First-Letter Sharding (Simple but Flawed)

Partition key = first_character_of_prefix Partitions: - Partition "a": all prefixes starting with "a" (e.g., "a", "ap", "app", "apple") - Partition "b": all prefixes starting with "b" - ... - Partition "z": all prefixes starting with "z" Storage per partition: ~50 GB (average) Query distribution: Highly skewed - Partition "a": 15% of all queries - Partition "b": 12% of all queries - Partition "z": 0.5% of all queries Problem: Letter "a" partition overloaded; "z" underutilized.

Approach 2: Consistent Hashing (Better)

Partition key = hash(prefix) mod 256 Benefits: - Distributes load evenly despite Zipfian queries - Can add/remove partitions; only 1/N data reshuffles - Hot prefixes scattered across partitions, reducing single-partition bottleneck Trade-offs: - Must hash each prefix; adds 1-2ms latency - Cache coherence harder (Redis cluster mode required) Ideal: 256 partitions across 32 servers = 8 partitions per server.

Approach 3: Hybrid (Recommended)

Strategy Details
Hot set: 1-3 char prefixes Keep fully cached in Redis cluster (all nodes). ~3.4 GB memory. No hashing needed; always fast hit.
Warm set: 4 char prefixes Consistent hash across 64 partitions. Cache misses fetch from Cassandra; write to local Redis replica.
Cold set: 5+ char prefixes On-demand Cassandra lookups. Don't cache (too many combinations). Rare queries anyway.

Virtual Nodes for Load Balancing

Instead of single partition per node, use virtual nodes: - Physical server: "suggestion-server-01" - Virtual partitions: [vn-0, vn-1, vn-2, ..., vn-7] (8 vnodes per physical node) - When adding server: move 8 vnodes from existing servers; only 8/256 data reshuffles Benefits: - Smooth scaling - Better load distribution - Fault tolerance: if one server fails, load spreads across multiple others
Warning: Avoid naive sharding by first-letter. "a" partition becomes hot, maxing out CPU/network. Use consistent hashing with virtual nodes to distribute load.

Personalization Layer

Design: Global Suggestions + Per-User Overlay

Global suggestions (from global trie/cache): Input: "ap" Output: ["apple", "app", "api", "apply", ...] (global top-10) Per-user overlay (user history): User has searched: "apple pie", "apple watch", "apple store" User's boosted: ["apple", "apple watch", "apple pie", ...] Blended result (weighted merge): score_final = 0.7 × global_score + 0.3 × user_score Return: Merged top-10 by final score

Implementation

def get_suggestions(prefix: str, user_id: str) -> List[Suggestion]: # 1. Get global suggestions (from cache) global_sugg = redis_get(f"{lang}:{prefix}") # O(1) if not global_sugg: global_sugg = cassandra_trie_lookup(prefix) # O(m) # 2. Get user's search history (optional; only if user_id provided) user_history = redis_get(f"user_history:{user_id}") # TTL: 30 days user_suggestions = filter_by_prefix(user_history, prefix) # 3. Blend scores blended = {} for term, global_score in global_sugg: user_score = get_user_score(term, user_suggestions) blended[term] = 0.7 × global_score + 0.3 × user_score # 4. Sort and return top-10 return sorted(blended.items(), key=lambda x: -x[1])[:10]

Privacy Considerations

Concern Mitigation
User search history stored server-side Hash user_id → anonymized ID before storing. Use client-side storage (IndexedDB) for non-sensitive history.
History leakage if Redis compromised Encrypt search history in Redis. Use TLS 1.3 for all transit.

Multi-Language Support

Challenge: Unicode Handling & Separate Tries

Different languages require different ranking (English queries dominate US users, Spanish for Latin America). Unicode normalization is critical: "café", "cafè", "cafe" should match the same token.

Architecture: Separate Trie per Language

Storage per language: - English (en): 5M terms, 500 GB - Spanish (es): 3M terms, 300 GB - French (fr): 2M terms, 200 GB - German (de): 2M terms, 200 GB - Chinese (zh): 1M terms, 100 GB (fewer terms due to complexity) - Total: ~1.3 TB (highly compressible) Cassandra schema: CREATE TABLE trie_snapshots ( version TEXT, -- "v202403" language_code TEXT, -- "en", "es", "fr", "de", "zh" partition_key INT, trie_binary BLOB, ... ) Redis naming: - en:ap → English prefix "ap" - es:ap → Spanish prefix "ap" - fr:ap → French prefix "ap"

Unicode Normalization

import unicodedata def normalize_query(query: str, language: str) -> str: # 1. Unicode normalization (NFD = decomposed) normalized = unicodedata.normalize('NFD', query.lower()) # 2. Remove accents normalized = ''.join( c for c in normalized if unicodedata.category(c) != 'Mn' # Mn = mark, non-spacing ) # 3. Language-specific rules if language == 'de': normalized = normalized.replace('ü', 'ue') # German: ü → ue return normalized # Examples: # normalize_query("Café", "en") → "cafe" # normalize_query("Straße", "de") → "strasse" (ß → ss)

Language Detection & Routing

User's language inferred from: (1) browser Accept-Language header, (2) user account settings, (3) IP geolocation. API parameter: ?lang=en. Service routes to appropriate Cassandra partition + Redis keys.

Trade-offs & Alternatives

Trie as core DS ✅ Trie Sorted list, suffix tree, hash + set Trie: O(m) exact match; O(1) top-10 return. Sorted list O(log n) per query (slower). Suffix tree overkill.
Ranking algorithm ✅ Weighted blend
(popularity + recency)		 Pure frequency, time-decay only, ML model Pure frequency: misses trends. Time-decay: ignores legacy popularity. ML: too slow at inference time (100ms SLA).

Persistent storage ✅ Cassandra HBase, DynamoDB, PostgreSQL HBase: requires Hadoop; operational burden. DynamoDB: expensive at 5B QPS. PostgreSQL: not write-optimized for weekly full trie uploads.
Trie update freq ✅ Weekly full rebuild Daily incremental, real-time updates Daily: high compute cost. Real-time: requires distributed trie merges; complexity spike.

(hot set global)		 First-letter sharding, range sharding First-letter: "a" hotspot overloaded. Range sharding: similar hotspot issues.
Personalization scope ✅ Per-user overlay
(blended scores)		 Pure global (no personalization), per-user full ranking No personalization: less relevant for users. Full per-user ranking: too slow (real-time merge expensive).

Decision Framework: Why These Choices?

  1. Scalability: Trie + cache hits handle 100K QPS with <100ms latency. Simpler structures fail at scale.
  2. Cost: Cassandra's LSM trees make weekly bulk writes efficient. Real-time updates would need Kafka consumers + Redis updates constantly; expensive.
  3. Operational Simplicity: Weekly batch trie rebuild is predictable, easy to monitor, and trivial to rollback (blue-green).
  4. Accuracy: Weighted blend of popularity + recency is interpretable, fast, and good enough. ML models overkill for MVP.

Common Follow-Up Questions & Answers

Q: What if Redis fails?
A: Degrade gracefully. Cassandra trie stays hot in memory (pre-load on startup). Cache misses hit Cassandra; latency goes from 5ms → 30ms. Error rate stays 0% but P99 → 150ms. Acceptable SLA breach for hours until Redis recovers. Have automated alerts to page on-call; restore in 10 min via cluster failover.
Q: How do you handle typos (e.g., "appl" → "apple")?
A: Not in MVP. Requires expensive fuzzy matching (Levenshtein distance = O(m×n)). Only worth implementing if >5% of queries have typos (monitor in logs). If needed, pre-compute edit-distance neighbors during trie build; store alongside suggestions.
Q: What's the freshness of suggestions?
A: Weekly vocabulary (new terms added), daily scores (popularity updated). So a trending query from Monday hits trie by Sunday. Acceptable lag for autocomplete (user searches for old term, sees older popularity rank).
Q: How do you avoid cache stampede on hotkeys?
A: (1) Background refresh job: check TTLs on hot prefixes; refresh before expiry. (2) Probabilistic early expiry: when key TTL < 15 min, trigger async refresh. (3) Cache lock: first miss acquires lock, recomputes, others wait. Redis Redlock pattern.
Q: How do you support multi-language?
A: Separate trie per language (en, es, fr, de, zh). Cassandra key = (version, language_code, partition). Redis key = {language}:{prefix}. Detect language from browser Accept-Language header or user setting. Total storage ~1.3 TB for 6 languages.
Q: Why Cassandra over DynamoDB?
A: DynamoDB pricing: 5B queries/day → RCU/WCU costs $1M+/month. Cassandra self-hosted: commodity hardware, 3x replication on 30 nodes = $10-20K/month ops cost. Also, Cassandra LSM trees optimized for our weekly bulk trie writes (bulk loading). DynamoDB better for transactional workloads.
