Design a URL Shortener (bit.ly)

A comprehensive system design for a distributed URL shortening service at tech-lead/staff-engineer level. Handles 100M new URLs per day with 100:1 read ratio.

Overview

A URL shortener is a service that converts long URLs into short, unique, and easy-to-share identifiers. Users submit a long URL, receive a short alias (e.g., bit.ly/abc123), and when accessed, are redirected to the original URL.

This design targets a large-scale service similar to bit.ly, operating at:

  • Write Load: 100M new URLs/day = ~1,200 writes/sec
  • Read Load: 100:1 ratio = ~120,000 reads/sec
  • Data Retention: 5 years = ~150TB storage
  • Cache Hit Rate: 80% (Pareto distribution on URL popularity)

Functional Requirements

Core Features

  • Shorten URL: Accept a long URL and return a short, unique code
  • Redirect: Given a short code, redirect to the original URL with 301 status
  • Custom Alias: Allow users to reserve custom vanity URLs (e.g., bit.ly/mycompany)
  • Analytics: Track clicks, unique visitors, referrers, device types, geography
  • Expiry: URLs expire after a configurable time (default 1 year); return 410 Gone
  • Delete: Users can manually delete their shortened URLs
  • Private URLs: Optional password protection or authorization check

Edge Cases

  • Collision: Two users request the same custom alias → return 409 Conflict
  • Invalid/Expired URL: Return 404 Not Found or 410 Gone appropriately
  • Idempotency: Shortening the same long URL twice should return the same short code
  • Malicious URLs: Detect and block URLs flagged by Google Safe Browsing
  • Rate Limiting: Prevent abuse (e.g., 100 URL shortens per minute per user)
  • URL Preview: Show target URL (domain) before redirect for security
Tip: Idempotency is crucial in distributed systems. Use (user_id, long_url) as a unique key to detect duplicates and return cached results.

Cache Estimation (Redis)

Assuming 80% cache hit rate and Zipfian distribution (top 20% URLs generate 80% of reads):

  • Unique URLs per day: 100M
  • Daily active URLs: ~500M (rolling window)
  • Hot set (top 20% by traffic): 100M URLs
  • Per cache entry: ~2.1 KB
  • Cache size needed: 100M × 2.1 KB = 210 GB
  • With replication: ~500 GB (3x replication across cache nodes)
Insight: The cache footprint is dominated by the hot set. Redis cluster with 500GB across 5 nodes (100GB each) provides sufficient capacity with room for growth. Use consistent hashing to balance load.

Network Estimation

  • Read payload (HTTP 301 redirect): ~500 bytes × 120K QPS = 60 Gbps (before cache hit)
  • With 80% cache hit: 60 Gbps × 0.2 = 12 Gbps to DB (manageable)
  • Write payload: ~2 KB × 1.2K QPS = 2.4 Mbps (negligible)

High-Level Architecture

The URL shortener follows a classic layered architecture optimized for reads and eventual consistency.

Component Overview

  • Load Balancer: Distributes incoming requests (nginx, AWS ELB) to read/write services
  • Read Service: Handles redirect requests; checks Redis cache first, queries DB on miss
  • Write Service: Shortens URLs; reserves key from KGS, stores in DB, populates cache
  • KGS (Key Generation Service): Pre-generates and reserves unique short codes; prevents collisions
  • Redis Cache: In-memory cache of hot URLs (80/20 rule); 500GB cluster for 100M hot URLs
  • MySQL/NoSQL DB: Persistent storage of URL mappings; sharded by hash for scalability
  • Cassandra: Time-series database for analytics (clicks, unique visitors, timestamps)
  • CDN: Caches top 20% URLs at edge locations for ultra-low latency redirects
Warning: Never use a single database. Sharding is mandatory at this scale. Use consistent hashing on the short code to distribute URLs evenly.

Key Generation Strategy

Approach Comparison

MD5 + Truncation

  • Hash long URL → take first 7 chars
  • Fast, deterministic
  • Problem: Collisions on hash truncation (~0.1% collision rate for 7 chars)
  • Problem: Predictable; can deduce original URL

Base62 Encoding

  • Hash long URL → encode in Base62
  • Better distribution than MD5 truncation
  • Problem: Still deterministic; security risk
  • Problem: Collisions still occur with truncation

Random Base62 (No Collision Detection)

  • Generate random 7-char Base62 string
  • Fast, non-deterministic
  • Problem: Requires DB collision detection
  • Problem: Retry loop on collision (expensive)

KGS (Key Generation Service) ✓ RECOMMENDED

  • Dedicated service pre-generates keys
  • No runtime collisions, no retry loops
  • Supports custom aliases
  • Advantage: Horizontal scaling via standby KGS

KGS Design (Recommended)

How it works:

  1. KGS maintains a pool of pre-generated unique keys (e.g., 1M keys available)
  2. Write Service requests a key from KGS (reserved immediately)
  3. KGS marks key as reserved; returns to Write Service
  4. Write Service stores URL in DB with reserved key
  5. If DB write fails, key is released back to KGS pool

Key generation algorithm (Base62):

BASE62_CHARSET = "0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz"

def encode_base62(num):
    """Convert integer to Base62 string (7 chars max = 62^7 unique keys)"""
    if num == 0:
        return "0"
    digits = []
    while num > 0:
        digits.append(BASE62_CHARSET[num % 62])
        num //= 62
    return ''.join(reversed(digits))

def decode_base62(s):
    """Convert Base62 string to integer"""
    result = 0
    for char in s:
        result = result * 62 + BASE62_CHARSET.index(char)
    return result

# Example: 62^7 = ~3.6 trillion unique 7-char keys
# More than enough for our 180B URL requirement over 5 years
print(encode_base62(12345))  # Output: "3D7"
print(decode_base62("3D7"))  # Output: 12345

KGS Architecture

  • Data Storage: Two ranges in Zookeeper (or etcd): [available, reserved]
  • Pre-generation: Background thread generates keys in batches of 1M
  • Reservation: Atomic compare-and-swap when Write Service requests a key
  • High Availability: Primary + Standby KGS; standby takes over if primary fails
  • Monitoring: Alert if pool < 100K keys (indicates generation lag)
Note: KGS can handle collisions gracefully: if a custom alias is already taken, it fails the request before DB write. KGS only manages auto-generated keys; custom aliases bypass KGS and hit the DB directly (with collision detection).

Sharding Strategy

Shard Key: Use the hash (short code) as the partition key with consistent hashing.

# Consistent hashing example (Python)
import hashlib

class ConsistentHash:
    def __init__(self, num_shards=16):
        self.num_shards = num_shards
        self.ring = {}
        self._build_ring()

    def _build_ring(self):
        for shard_id in range(self.num_shards):
            hash_value = int(hashlib.md5(f"shard-{shard_id}".encode()).hexdigest(), 16)
            self.ring[hash_value] = shard_id

    def get_shard(self, hash_key):
        """Determine which shard a hash belongs to"""
        key_hash = int(hashlib.md5(hash_key.encode()).hexdigest(), 16)
        sorted_keys = sorted(self.ring.keys())
        for ring_hash in sorted_keys:
            if key_hash <= ring_hash:
                return self.ring[ring_hash]
        return self.ring[sorted_keys[0]]  # Wrap around

# Example usage
ch = ConsistentHash(num_shards=16)
print(ch.get_shard("abc123"))  # Output: 7 (shard ID)
print(ch.get_shard("xyz789"))  # Output: 3

Shard Distribution:

  • 16 shards initially; each shard handles ~7.5B URLs over 5 years
  • Each shard: single MySQL instance + 2 replicas for HA
  • Add/remove shards with consistent hashing (gradual rebalancing)
Insight: Avoid sharding by user_id. User often wants to list all their URLs; cross-shard queries are expensive. Instead, use hash as shard key and maintain a secondary index (user_id → list of hashes) in Redis.

Write Path Deep Dive

Step-by-Step Process

  1. Validate Input: Check long URL format, length limits, blacklist (malware detection)
  2. Check Idempotency: Hash(user_id + long_url) → query cache/DB for existing entry
  3. Validate Custom Alias (if provided): Check if alias is available; return 409 Conflict if taken
  4. Reserve Key from KGS: Request a unique short code; KGS returns immediately (low latency)
  5. Insert into MySQL: Write (hash, long_url, user_id, expires_at) to sharded DB
  6. Populate Cache: Insert into Redis with TTL matching expiry
  7. Return Response: HTTP 201 Created with shortened URL and metadata

Collision Handling

def shorten_url(user_id, long_url, custom_alias=None):
    """Shorten a URL with collision detection and idempotency"""

    # Step 1: Idempotency check
    idempotency_key = hash_md5(f"{user_id}:{long_url}")
    cached = redis.get(f"idempotency:{idempotency_key}")
    if cached:
        return cached  # Return cached short URL

    # Step 2: Check custom alias availability
    if custom_alias:
        if db.exists(alias=custom_alias):
            raise ConflictError(f"Alias {custom_alias} already taken")
        hash_key = custom_alias
    else:
        # Step 3: Reserve key from KGS
        hash_key = kgs.reserve_key()

    # Step 4: Attempt DB write
    try:
        db.insert_url(
            hash=hash_key,
            original_url=long_url,
            user_id=user_id,
            alias=custom_alias,
            expires_at=datetime.now() + timedelta(days=365)
        )
    except IntegrityError:
        # Collision! Release key to KGS and retry
        if not custom_alias:
            kgs.release_key(hash_key)
        return shorten_url(user_id, long_url, custom_alias)

    # Step 5: Populate cache
    result = {
        "short_url": f"bit.ly/{hash_key}",
        "hash": hash_key,
        "expires_at": ...
    }
    redis.set(f"idempotency:{idempotency_key}", json.dumps(result), ex=86400)
    redis.set(f"url:{hash_key}", long_url, ex=365*86400)

    return result

Redirect Logic

def redirect_url(hash_key):
    """Redirect to original URL with cache-aside pattern"""

    # Step 1: Check CDN (edge location)
    # (CDN automatically handles; if CDN has cached redirect, returns directly)

    # Step 2: Check Redis cache
    cached_url = redis.get(f"url:{hash_key}")
    if cached_url:
        # Update analytics (async)
        kafka_producer.send("url_clicks", {
            "hash": hash_key,
            "timestamp": datetime.now(),
            "ip": request.remote_addr,
            "user_agent": request.headers['User-Agent'],
            "referrer": request.referrer
        })
        # Return HTTP 301 (permanent redirect)
        return redirect(cached_url, code=301)

    # Step 3: Query MySQL (cache miss)
    shard_id = consistent_hash.get_shard(hash_key)
    db_connection = mysql_pool.get_connection(shard_id)
    url_record = db_connection.query(
        "SELECT original_url, expires_at, is_password_protected, password_hash FROM urls WHERE hash = %s",
        (hash_key,)
    )

    if not url_record:
        return error_response(404, "URL not found")

    original_url, expires_at, is_protected, pwd_hash = url_record

    # Step 4: Check expiry
    if datetime.now() > expires_at:
        return error_response(410, "URL has expired")

    # Step 5: Check password protection
    if is_protected:
        provided_pwd = request.args.get('pwd')
        if not provided_pwd or not bcrypt.verify(provided_pwd, pwd_hash):
            return error_response(401, "Password required")

    # Step 6: Populate cache
    redis.set(f"url:{hash_key}", original_url, ex=(expires_at - datetime.now()).total_seconds())

    # Step 7: Track analytics (async)
    kafka_producer.send("url_clicks", {...})

    # Step 8: Return redirect
    return redirect(original_url, code=301)

301 vs. 302 Trade-off

Aspect HTTP 301 (Permanent) HTTP 302 (Temporary)
Browser Caching Cached indefinitely; reduces load Not cached; every request hits us
SEO Passes SEO authority to target Retains authority at shortener
Update URL Browser cache issue (need cache clear) Easy; immediate effect
Recommendation ✓ Use for public/stable URLs Use for A/B tests, dynamic redirects

Recommendation: Use 301 (Permanent) by default for maximum caching benefit. Allow users to opt-in to 302 for mutable URLs.

Cache-Aside Pattern

def get_url_with_cache(hash_key):
    """Cache-aside pattern (lazy loading)"""

    # Check cache first
    value = redis.get(hash_key)
    if value is not None:
        return value  # Cache hit

    # Cache miss: fetch from DB
    value = db.get(hash_key)

    # Populate cache for future requests
    redis.set(hash_key, value, ex=3600)  # 1 hour TTL

    return value
Insight: Use HTTP 301 redirects at the CDN level. Most users (80%) will hit the CDN, which caches the redirect response. This reduces load on our origin by 80%.

Analytics Pipeline

Architecture

  • Event Capture: Every redirect generates a click event (non-blocking, fire-and-forget)
  • Message Queue: Kafka topic url_clicks (partition by hash for ordering)

  • Time-Series Database: Cassandra stores aggregated metrics (clicks, unique visitors, device breakdown)
  • Cache: Redis cache for real-time dashboard (hot reads)

Aggregation Pipeline (Flink)

def process_click_events(events_stream):
    """Aggregate clicks into 1-minute windows (low latency)"""

    # Window by 1 minute, partition by hash
    aggregated = (
        events_stream
        .filter(lambda e: e.hash)  # Remove invalid events
        .window(TumblingEventTimeWindows.of(Time.minutes(1)))
        .keyBy(lambda e: e.hash)
        .aggregate(
            init_fn=lambda: {
                "total_clicks": 0,
                "unique_ips": set(),
                "device_breakdown": defaultdict(int),
                "top_referrers": defaultdict(int),
                "geography": defaultdict(int)
            },
            agg_fn=lambda state, event: {
                "total_clicks": state["total_clicks"] + 1,
                "unique_ips": state["unique_ips"] | {event.ip_address},
                "device_breakdown": {
                    **state["device_breakdown"],
                    event.device_type: state["device_breakdown"].get(event.device_type, 0) + 1
                },
                "top_referrers": {
                    **state["top_referrers"],
                    event.referrer: state["top_referrers"].get(event.referrer, 0) + 1
                },
                "geography": {
                    **state["geography"],
                    event.country_code: state["geography"].get(event.country_code, 0) + 1
                }
            }
        )
    )

    # Write to Cassandra (time-series)
    aggregated.add_sink(
        CassandraSink(
            keyspace="analytics",
            table="click_metrics",
            session=cassandra_session
        )
    )

    # Also write hot data to Redis for dashboard
    aggregated.map(lambda x: redis.set(f"metrics:{x.hash}", json.dumps(x)))

URL Expiry & Cleanup

Expiry Strategy

Lazy Expiry (Recommended):

  • Don't actively delete expired URLs from the DB
  • When accessed, check expires_at timestamp
  • If expired, return 410 Gone without touching the DB
  • Overhead: single timestamp comparison per read (negligible)

Background Cleanup Job (Optional):

  • Run daily; deletes URLs older than 1 year + grace period (2 weeks)
  • Reclaim storage space; archive expired URLs to S3
  • Use cursor-based pagination to avoid full table scans

Cleanup Job Implementation

def cleanup_expired_urls(grace_period_days=14):
    """Background job to delete expired URLs and archive to S3"""

    cursor = 0
    batch_size = 10000
    cutoff_date = datetime.now() - timedelta(days=1+grace_period_days)

    while True:
        # Cursor-based pagination (avoids offset performance cliff)
        results = db.query("""
            SELECT id, hash, original_url, click_count FROM urls
            WHERE expires_at < %s AND id > %s
            ORDER BY id
            LIMIT %s
        """, (cutoff_date, cursor, batch_size))

        if not results:
            break

        # Archive to S3 (for compliance/analytics)
        archive_data = json.dumps([
            {
                "hash": r.hash,
                "original_url": r.original_url,
                "click_count": r.click_count,
                "expired_at": r.expires_at.isoformat()
            }
            for r in results
        ])

        archive_key = f"s3://url-shortener-archive/{cutoff_date.isoformat()}/{cursor}.json.gz"
        s3_client.put_object(Bucket="url-shortener-archive", Key=archive_key, Body=gzip.compress(archive_data))

        # Delete from MySQL
        ids_to_delete = [r.id for r in results]
        db.execute(f"DELETE FROM urls WHERE id IN ({','.join(['%s']*len(ids_to_delete))})", ids_to_delete)

        # Delete from Redis cache
        for r in results:
            redis.delete(f"url:{r.hash}")
            redis.delete(f"metrics:{r.hash}")

        cursor = max([r.id for r in results])
        logger.info(f"Deleted {len(results)} expired URLs; next cursor: {cursor}")

Security Considerations

  • Google Safe Browsing API: Query for each new URL; block if flagged for phishing/malware

  • DNSBL Lookup: Check IP reputation; block known spam sources
  • User Reports: Allow users to report malicious URLs; queue for manual review

  • Per-User Rate Limit: 100 URLs/minute (token bucket in Redis)
  • Per-IP Rate Limit: 500 requests/minute (detect bot sweeping)

  • CloudFlare/AWS Shield: Distributed DDoS protection at CDN edge

Password-Protected & Private URLs

def create_private_url(user_id, long_url, password=None):
    """Create password-protected URL"""

    hash_key = kgs.reserve_key()

    # Hash password with bcrypt
    password_hash = None
    if password:
        password_hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt())

    db.insert_url(
        hash=hash_key,
        original_url=long_url,
        user_id=user_id,
        is_password_protected=bool(password),
        password_hash=password_hash,
        is_private=True
    )

    return {"short_url": f"bit.ly/{hash_key}?pwd=required"}

# On redirect:
def redirect_private_url(hash_key, password=None):
    """Redirect after password check"""

    url_record = db.get_url(hash_key)

    if url_record.is_password_protected:
        if not password or not bcrypt.checkpw(password.encode(), url_record.password_hash):
            return error_response(401, "Incorrect password")

    # User authenticated; return redirect
    return redirect(url_record.original_url, code=301)

SQL Injection Prevention

  • Parameterized Queries: Always use prepared statements; never concatenate user input
  • ORM: Use SQLAlchemy/Hibernate; automatic parameterization
  • Input Validation: Hash must be 7 alphanumeric; reject invalid chars

XSS Prevention

  • URL Encoding: Encode long URL in HTTP response to prevent XSS
  • Content Security Policy: Set CSP header; restrict script execution
  • httpOnly Cookies: Never store sensitive data in client-side cookies

Data Privacy & GDPR Compliance

  • Right to Deletion: Users can delete all their URLs + analytics
  • Data Minimization: Store only necessary fields (hash, URL, timestamps)
  • Encryption at Rest: AES-256 for sensitive data (passwords, IPs)
  • Encryption in Transit: TLS 1.3 for all API requests
  • IP Anonymization: Truncate last octet of IPv4 addresses in logs
Insight: Implement a "URL safety score" based on Google Safe Browsing, domain reputation, and user feedback. Show users a warning if score < 0.5.

Scalability Deep Dive

Database Sharding Strategy

Problem: Single DB can't handle 1.2K writes/sec or store 150TB. Solution: Shard by hash.

  • 16 shards initially; each MySQL instance has 3TB per year
  • Consistent hashing allows adding shards without full migration
  • Shard rebalancing: move 50% of keys from shard N to shard N+1
  • Cost: ~16 MySQL instances × 3 replicas = 48 total DB instances

KGS Failure Modes & Recovery

Scenario Impact Mitigation
Primary KGS crashes Can't generate new keys Standby KGS takes over in <1 sec (Zookeeper leader election)
Key pool depleted Can't reserve keys; requests fail Alert if pool < 100K; background generation ramps up
KGS generates duplicate keys DB insert fails; retry loop KGS uses atomic counter; impossible to generate duplicates
Network partition KGS unreachable Fallback: client generates random key, retries on collision

Cache Scalability

  • Redis Cluster: 500 GB across 5 nodes (100 GB each)
  • Consistent Hashing: Balance cache entries across nodes
  • Replica Factor: 2 (primary + 1 replica); survives 1 node failure
  • Sentinel: Automatic failover if node goes down
  • Eviction Policy: LRU (least recently used) when full

Write Service Horizontal Scaling

# Load balancer (nginx) configuration
upstream write_service {
    least_conn;  # Route to server with fewest active connections
    server write-service-1:8080;
    server write-service-2:8080;
    server write-service-3:8080;
    server write-service-4:8080;
    keepalive 32;  # Connection pooling
}

# Each service instance:
# - Connects to KGS (single endpoint, but highly available)
# - Connects to sharded MySQL (pool of 16 connections, one per shard)
# - Connects to Redis cache (cluster aware client)

Read Service Horizontal Scaling

  • Stateless design: No affinity needed; requests can hit any instance
  • Load balancer: Round-robin or least-conn across 10+ instances
  • Connection pooling: Each instance maintains 1 Redis cluster connection, 1 MySQL connection pool
  • DNS caching: Each instance caches shard node addresses; TTL 5 minutes
Tip: Use auto-scaling groups in AWS. Monitor p99 latency; add instances if p99 > 200ms. Remove instances if p99 < 50ms for 5 minutes.

Failure Modes & Mitigations

Failure Mode Severity Impact Mitigation
MySQL shard down High Can't read/write for 1/16 of URLs Replicas take over; automatic failover via MHA
Redis cluster down High 20% latency increase; 500 Gbps→ 12 Gbps to DB Fallback to DB; rebuild cache from DB on recovery
KGS standby also down Critical Can't generate new URLs Clients fallback to random key + collision retry; degrades to ~50% success rate
Network partition (KGS unreachable) Medium New URL creation fails Implement timeout + fallback; circuit breaker pattern
Cascade: Cache miss + slow DB High Read latency spikes (p99 > 5 sec) Implement read timeout (500ms); return 503 Service Unavailable if exceeded
Storage full High Can't write new URLs Monitoring alerts; add capacity before reaching threshold
Data corruption in MySQL Critical URLs become unresolvable Binary log replication; restore from backup to another shard

Disaster Recovery Plan

  • RTO (Recovery Time Objective): < 5 minutes
  • RPO (Recovery Point Objective): < 1 minute (binlog replication)
  • Multi-region backup: Replicate DB binlogs to another region every minute
  • Point-in-time restore: Restore from backup + replay binlogs to any point
  • Testing: Monthly DR drills; verify restores work
Warning: KGS is a single point of failure for writes. Always have a standby ready. Even with fallback, you lose the guarantee of no collisions.

Trade-offs & Design Decisions

Decision Pro Con Why Chosen
KGS (vs random collision retry) No collision retries; fast writes Operational complexity; single point of failure At 1.2K writes/sec, collision retry loop is unacceptable latency
Cache-aside (vs write-through) Lower latency; eventual consistency Cache misses possible; stale data on update Read-heavy workload; cache-aside minimizes write latency
HTTP 301 (vs 302) Browser caches; reduces load 80% Can't change target URL; users see cached version URLs are immutable by design; 301 is correct
Lazy expiry (vs background cleanup) Simpler code; no background jobs Expired URLs still consume storage Lazy expiry is free on read; cleanup is optional optimization
Sharding by hash (vs user_id) Even distribution; handles hotspot users Hard to list user's URLs (requires scatter-gather) Read-heavy; hash-based sharding minimizes skew
Cassandra for analytics (vs time-series index in MySQL) Better write throughput; built for time-series Another system to operate Analytics writes are async; Cassandra is cheaper at scale
Base62 encoding (vs MD5 truncation) Better distribution; no SQL injection risk Non-deterministic; longer code possible Deterministic hashes reveal URL structure; non-deterministic is safer

Common Follow-Up Questions & Answers

Q: How to handle custom aliases collisions?

A: Check if alias exists in DB before write. If taken, return 409 Conflict. Allow users to reserve vanity URLs (paid feature).

Q: How to ensure no duplicate URLs?

A: Hash (user_id + long_url) → unique constraint. On conflict, return cached short URL instead of creating duplicate.

Q: What if a URL is updated (redirect target changes)?

A: URLs are immutable by design. Users must delete and re-create. This allows HTTP 301 caching and SEO pass-through.

Q: How to handle malicious/spam URLs?

A: Query Google Safe Browsing API on write. Block flagged URLs. Allow user reports; queue for manual review. Show domain preview before redirect.

Q: Can KGS be a bottleneck?

A: No; KGS is a single RPC call (~5ms). Sharding KGS by range is possible: KGS-1 generates [0, 62^7/2), KGS-2 generates [62^7/2, 62^7). Standby KGS for HA.

Q: How to list all URLs for a user?

A: Secondary index: user_id → [hash1, hash2, ...] in Redis. Cache miss: scan MySQL shards in parallel (expensive). Alternative: store user's URLs in a separate user_urls table.

Q: How to handle geographic distribution (CDN)?

A: Replicate read service to edge locations. Cache redirects (301 + Location header) at CDN. DynamoDB Global Tables for metadata if needed.

Q: Can you share DB shards across regions?

A: No; shards should be co-located for latency. Each region has its own 16 shards. Master-master replication between regions with conflict resolution (last-write-wins).

Common Mistakes to Avoid

  • ❌ Using MySQL AUTO_INCREMENT as hash: Predictable (sequential); easy to enumerate all URLs
  • ❌ Storing raw IP addresses: Privacy issue; anonymize (truncate last octet)
  • ❌ Deleting expired URLs immediately: Breaks links; use lazy expiry + S3 archive
  • ❌ Single KGS instance: Single point of failure; always have standby
  • ❌ Using 302 redirects by default: Every request hits origin; use 301 for immutable URLs
  • ❌ Sharding by user_id: Breaks "list user's URLs" queries; use hash-based sharding
  • ❌ Synchronous analytics: Blocks redirect; always make async (Kafka)
Insight: Interviewers care more about how you think through trade-offs than the specific numbers. Justify every decision. If you say "use Redis cache," explain why (80/20 rule, read-heavy, cache-aside pattern). If you say "shard by hash," explain why not user_id.
