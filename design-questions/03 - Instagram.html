Design Instagram

A comprehensive system design for a photo-sharing social platform at scale.

Overview

Instagram is a photo-sharing and social networking platform where users can upload photos, like, comment, follow other users, and discover content through a personalized feed. The design must handle massive scale (500M daily active users), real-time interactions, and extreme spiky load on trending content.

Core user flows:

  • Upload a photo with caption, location, tags
  • View personalized feed of recent posts from followed users
  • Like, comment, share posts
  • View Stories (24-hour ephemeral content)
  • Discover content via Explore, hashtag search, location
  • Receive real-time notifications (likes, comments, follows)
  • Handle direct messaging (out of scope for this deep-dive)

This document focuses on the feed, upload, and notifications pipelines at 500M DAU scale.

Functional Requirements

  1. Photo Upload: User uploads photo + caption, location, tags. System assigns unique ID, stores in S3, generates thumbnails/previews.
  2. Feed: User sees chronological or ranked feed of posts from followed users. Real-time updates when friends post.
  3. Engagement: Users can like, unlike, comment, delete comment, share posts. Counts are accurate.
  4. Follow/Unfollow: Users can follow/unfollow others. Feed updates reflect new follows immediately.
  5. Stories: Ephemeral content (24-hour TTL). User can view story, see who viewed it, reply to story.
  6. Search & Discover: Search users, hashtags, locations. Trending explore page personalized by interests.
  7. Notifications: Real-time notifications for likes, comments, follows, mentions. Aggregation (3 people liked your photo).
  8. Profile: View profile, bio, photo grid, follower/following counts, stats.
  9. Edge Cases:
    • Celebrity with 100M followers cannot fan-out to all in-memory. Lazy-load on read.
    • Duplicate upload detection (same photo, different metadata).
    • Rate-limiting on uploads (max 10 photos/hour/user).
    • Deleted content should be hidden from feed in <1 second.
    • Concurrent likes: counter must be strongly consistent (no race conditions).

High-Level Architecture

Diagram 1: High-Level System Architecture

Client Tier

Mobile apps (iOS, Android) and web clients connect over HTTPS. All requests are authenticated via OAuth 2.0 tokens passed in the Authorization header. Request versioning is handled via API headers to support backward compatibility.

Edge Tier

CDN: CloudFront or Akamai distribute static assets (JS, CSS, images) and cached API responses globally. Photos and thumbnails are cached at edge for <1s latency. TTL varies: trending content 5min, old content 1 hour.

Load Balancer: AWS NLB or HAProxy distributes traffic across API servers in multiple AZs. Health checks every 5 seconds, removes unhealthy instances.

API Tier

API Gateway: Routes requests to microservices. Implements rate-limiting (10 req/sec per user), request validation, logging, tracing (X-Ray).

Auth Service: Validates JWT tokens, handles session management, OAuth provider integration. Cached in Redis for <5ms latency.

Service Tier

Upload Service: Issues presigned S3 URLs, triggers media processing pipeline. Handles duplicate detection.

Feed Service: Fetches personalized feed. Reads from Redis cache (hot path), falls back to DB for cache misses. Supports both timeline (chronological) and ranked feeds.

User Service: Profile, follow/unfollow, discovery. Uses graph database (Neo4j or Cassandra) for follow relationships.

Notification Service: Kafka consumer that aggregates and dispatches notifications via APNs/FCM.

Data Tier

MySQL: User accounts, profiles, auth credentials. Strong consistency required. Sharded by user_id.

Cassandra: Photos, likes, comments, follows (wide-column family). High write throughput, eventual consistency acceptable. Distributed across multiple datacenters.

Redis: Feed timeline cache (list of photo IDs), session tokens, rate-limiting counters. ~100ms TTL for feeds.

S3: Photo objects, backups. Versioning enabled. Cross-region replication for disaster recovery.

Elasticsearch: Inverted index on photo tags, captions, hashtags, user bios. For search and explore.

Kafka: Event bus. Photo uploaded, liked, commented, deleted, story viewed. Fan-out worker subscribes.

Photo Upload Pipeline

Diagram 2: Photo Upload Pipeline (Presigned URL + Async Processing)

Step-by-Step Flow

  1. Request Presigned URL: Client calls GET /upload/presigned-url with photo size, MIME type. Upload Service validates user quota (10 photos/hour), generates 15-minute presigned S3 URL.
  2. Direct S3 Upload: Client uploads photo directly to S3 using presigned URL. Avoids going through our servers; reduces latency and server load.
  3. S3 Event Notification: S3 publishes ObjectCreated event.
  4. Kafka Publishing: S3 event triggers Lambda that publishes to Kafka topic "photo_uploaded" with event: { photo_id, user_id, timestamp, key, size, mime_type }.
  5. Async Processing (Dashed):
    • Media Processor: Consumes Kafka, downloads from S3, transcodes to multiple formats (original, 1200x1200, 400x400 thumbnail), uploads back to S3 with different keys.
    • DB Write: Writes photo metadata to Cassandra (photo_id, user_id, caption, location, tags, timestamp, like_count=0, comment_count=0).
    • Search Indexing: Elasticsearch indexes photo tags, captions, location for search.
    • CDN Cache Population: Pre-warm CDN with thumbnail for quick discovery.
    • Fan-Out: Post Service reads caption and mentions; if it mentions celebrities or uses viral hashtags, notify discovery algorithm.

Presigned URL Code Snippet

// POST /upload/presigned-url
{
  "file_size": 5242880,  // 5 MB
  "mime_type": "image/jpeg"
}

// Response
{
  "upload_id": "photo_abc123",
  "presigned_url": "https://instagram-photos.s3.amazonaws.com/...",
  "expiration": 900,  // 15 minutes
  "s3_key": "photos/user_123/photo_abc123.jpg"
}

// Client then directly uploads:
PUT presigned_url with binary photo data
// Server receives S3 ObjectCreated event, processes asynchronously
Tip: Presigned URLs reduce server load and latency. Direct client-to-S3 upload = 100ms vs. 5s through proxy. Use content MD5 hash to detect corruption; validate after upload.

Duplicate Detection

Before returning presigned URL, check if user has already uploaded same photo in last 24 hours. Use perceptual hashing (pHash) of the image; store in Redis with 24-hour TTL. If match found, return existing photo_id and link to it instead of creating duplicate.

Error Handling

If media processor fails (corrupt file, unsupported format), publish to dead-letter queue and alert ops. User sees photo in feed after DB write (step 4), so processing failures don't block visibility—thumbnails may be missing but original is available.

Database Design

User Accounts (MySQL, Sharded by user_id)

CREATE TABLE users (
  user_id BIGINT PRIMARY KEY,
  username VARCHAR(30) UNIQUE NOT NULL,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255),
  bio TEXT,
  profile_pic_url VARCHAR(1024),
  created_at TIMESTAMP,
  updated_at TIMESTAMP,
  is_private BOOLEAN DEFAULT FALSE
);

CREATE INDEX idx_username ON users(username);
CREATE INDEX idx_email ON users(email);

Why MySQL? Requires strong consistency (auth, account details), moderate QPS (login <100k/sec), good ACID support. Sharded across 1024 partitions by user_id hash.

Photos (Cassandra, Partitioned by user_id, Clustered by timestamp DESC)

CREATE TABLE photos (
  user_id BIGINT,
  photo_id TIMEUUID,  // sortable UUID with embedded timestamp
  caption TEXT,
  location_id BIGINT,  // foreign key to locations table
  created_at TIMESTAMP,
  like_count BIGINT,
  comment_count BIGINT,
  PRIMARY KEY (user_id, photo_id)
) WITH CLUSTERING ORDER BY (photo_id DESC);

// Search index (global secondary index on hashtags)
CREATE TABLE photos_by_hashtag (
  hashtag VARCHAR(100),
  photo_id TIMEUUID,
  user_id BIGINT,
  created_at TIMESTAMP,
  PRIMARY KEY (hashtag, photo_id)
) WITH CLUSTERING ORDER BY (photo_id DESC);

Why Cassandra? Photos are immutable once posted (mostly). Cassandra excels at write-heavy, time-series data. TIMEUUID ensures natural time-ordering. Read by user_id (for profile page) is O(1) partition lookup. Distributed across 3 datacenters with RF=3.

Warning: Like count is denormalized into photos table for fast reads. It's eventually consistent. Background job updates it hourly from the likes table. Acceptable 1-hour stale reads on like count.

Follows (Cassandra Graph)

CREATE TABLE follows (
  follower_id BIGINT,
  followee_id BIGINT,
  created_at TIMESTAMP,
  PRIMARY KEY (follower_id, followee_id)
);

CREATE TABLE followers (
  user_id BIGINT,
  follower_id BIGINT,
  created_at TIMESTAMP,
  PRIMARY KEY (user_id, follower_id)
) WITH CLUSTERING ORDER BY (created_at DESC);

Two tables for efficient bidirectional queries: "who am I following?" and "who follows me?". Cassandra's wide-column family allows millions of followers per user efficiently.

Likes (Cassandra, Counter)

CREATE TABLE likes (
  photo_id TIMEUUID,
  user_id BIGINT,
  created_at TIMESTAMP,
  PRIMARY KEY (photo_id, user_id)
);

// Denormalized counter for quick access
CREATE TABLE photo_like_counts (
  photo_id TIMEUUID,
  count COUNTER,
  PRIMARY KEY (photo_id)
);

Strong Consistency Pattern: Cassandra counters are eventually consistent, but acceptable for like counts. For critical accuracy (prize eligibility), use a distributed counter (Redis + async flush to DB).

Comments (Cassandra)

CREATE TABLE comments (
  photo_id TIMEUUID,
  comment_id TIMEUUID,
  user_id BIGINT,
  text TEXT,
  created_at TIMESTAMP,
  PRIMARY KEY (photo_id, comment_id)
) WITH CLUSTERING ORDER BY (comment_id DESC);

Stored by photo_id for fast retrieval of comment thread. No secondary index on user_id (expensive). If need to fetch user's comments, scan from a separate view table or use Elasticsearch.

Sharding Strategy

Table Shard Key Reason Replication
Users user_id % 1024 Distribute evenly, queries by user_id are common 3 replicas (MySQL master-slave)
Photos user_id (partition key) User's profile page queries all photos by user_id 3 (Cassandra across AZs)
Follows follower_id Queries "who am I following" are per user 3 (Cassandra)
Likes photo_id (partition key) Queries "who liked this photo" are per photo 3 (Cassandra)

Indexing Strategy

  • MySQL: B+ tree on username, email for login; foreign key on user_id in related tables.
  • Cassandra: No secondary indexes (too slow). Instead, denormalize into separate tables (photos_by_hashtag, users_by_location).
  • Elasticsearch: Inverted index on all text fields (tags, caption, user bio) for full-text search.
  • Redis: Sorted set (ZSET) for trending hashtags (score = frequency, member = tag_name).

News Feed: Fan-Out Architecture

Diagram 3: Feed Generation - Celebrity Problem & Hybrid Approach

The Celebrity Problem

When a user with 100M followers posts:

  • Push approach (naive): Write photo_id to 100M Redis lists (one per follower). Each list write is O(1), but 100M × ~100 bytes = 10 GB RAM immediately consumed. Not feasible.
  • Pull approach: Store post in DB, then fetch on feed read. When user views feed, query all post creators they follow. Slow (hundreds of DB roundtrips per feed fetch).
  • Hybrid approach (recommended): Push for normal users (<1M followers), Pull for celebrities (>1M followers).

Hybrid Fan-Out: Low-Follower Push

When user with <1M followers posts:

  1. Kafka Event: photo_id, user_id published to "photo_posted" topic.
  2. Fan-Out Worker: Consumes event. Fetches all follower_ids from followers table (e.g., 10k followers). Asynchronously writes to Redis feed list for each: 
    LPUSH feed:follower_id:123 photo_id:abc
EXPIRE feed:follower_id:123 86400  // 24 hour TTL
  3. Timeline Cache: Each user has Redis list storing IDs of photos from followed users, ordered by recency. Max 1,000 items per list (oldest items fall off automatically).
  4. Feed Service: GET feed:user_id:123 returns photo_ids. Fetch metadata (caption, like_count, comment_count) from Cassandra in parallel, sort by engagement, return to client.

Celebrity Pull Strategy (>1M followers)

When a celebrity posts, we don't fan-out. Instead:

  1. Kafka Event + DB Write: photo_id stored in Cassandra under user_id. NOT distributed to follower feeds.
  2. At Read-Time: Feed Service queries Cassandra for all photos from followed users, merging results from both regular posts (in Redis) and celebrity posts (in Cassandra). Use merge algorithm (two pointers) to interleave chronologically.
  3. Caching: Merge result cached for 10 seconds in Redis to avoid repeated Cassandra hits.

Feed Fetch Algorithm

function getFeed(user_id, page=0) {
  // 1. Get pre-computed feed from Redis (regular users' posts)
  cached_ids = LRANGE feed:user_id page*20 (page+1)*20-1

  // 2. Get recent posts from followed celebrities (not in Redis)
  followed_users = queryDB("SELECT followee_id FROM follows WHERE follower_id = ?")
  celebrity_photos = []
  for each followee_id in followed_users:
    if follower_count[followee_id] > 1M:
      photos = queryCassandra("SELECT photo_id, created_at FROM photos WHERE user_id = ? LIMIT 10")
      celebrity_photos.extend(photos)

  // 3. Merge and rank
  all_photos = cached_ids + celebrity_photos
  ranked = rankByEngagement(all_photos)

  // 4. Fetch metadata in parallel
  metadata = parallelFetchFromCassandra(ranked_ids)

  return metadata
}

Complexity & Trade-offs

Approach Write Latency Read Latency Storage Best For
Pure Push 10s+ (fan-out millions) 1ms (Redis list lookup) Very high (100M × followers) Small communities, equal followers
Pure Pull 1ms (direct write) 500ms+ (query 1000s of users) Low Few users, rare reads
Hybrid (Recommended) 5s avg (fan-out <1M) 50ms (cached + merge) Moderate (tuned by threshold) Instagram-scale: power-law follower distribution
Insight: Instagram uses a threshold of ~1M followers to decide push vs. pull. This is calibrated empirically: storage cost of pushing to 1M followers (~100 MB RAM) is acceptable, but 100M followers is not. The threshold can be adjusted based on Redis capacity.

Feed Ranking

Ranking Signals & Engagement Score

Instagram uses ML-based ranking to maximize time-on-app and engagement. Core signals:

Signal Weight Why
Like probability 25% ML model predicts if user will like post
Comment probability 15% Indicates deep engagement
Save probability 10% User intends to revisit
Share probability 5% Strongest signal of virality
Recency 20% Newer posts ranked higher, with decay
Relationship strength 15% DMs, past interactions with creator
Creator quality 10% Creator's engagement rate, bot score

Engagement Score Formula

score = (like_prob × 0.25 + comment_prob × 0.15 + save_prob × 0.10 + share_prob × 0.05)
        × recency_decay(age_seconds)
        × (1 + relationship_strength × 0.15)
        × (1 + creator_quality × 0.10)

Recency Decay Function

def recency_decay(age_seconds):
    """Exponential decay: recent posts score higher, age-off over 7 days"""
    half_life = 3600 * 24  // 1 day
    decay_factor = 2 ** (-age_seconds / half_life)
    return decay_factor

// Example: 1-day-old post scores 50% of current post
// 7-day-old post scores ~0.8% (effectively invisible)

ML Ranking Service

The ranking service is a separate microservice that:

  • Runs a neural net model (trained on millions of user interactions). Input: user features (age, location, interests), post features (tags, caption length, creator), interaction features (past likes, comments).
  • Outputs probability of like/comment/save/share.
  • Called at read-time for each user's feed. Scores all candidate posts (up to 1,000) and reranks top 20-50.
  • Model inference <50ms (served via TensorFlow Serving or ONNX).
  • Model trained weekly on accumulated engagement data (Kafka → Spark → feature store → training pipeline).

Fallback: Chronological Feed

If ranking service is down or model inference errors, fall back to chronological order. Acceptable latency degradation but preserves availability.

A/B Testing Infrastructure

Ranking changes are A/B tested:

  • Allocate 1-5% of users to new ranking algorithm.
  • Track metrics: session length, likes/session, shares, daily active return rate.
  • Statistically significant improvement (p < 0.05) required before full rollout.
  • Store experiment assignment in Redis (user_id → experiment_id, cached).

Stories Architecture

Overview

Stories are ephemeral content with 24-hour TTL. Unlike photos (permanent), stories disappear after 24 hours and are separate from the main feed. Key challenges:

  • Massive write volume (users can post 10+ stories/day).
  • Fast expiration (24-hour deletion).
  • View receipts (who viewed my story, at what time).
  • Ordered delivery (stories in order of poster, newest first).

Schema (Cassandra)

CREATE TABLE stories (
  user_id BIGINT,
  story_id TIMEUUID,  // Sortable by time
  media_url VARCHAR(1024),
  caption TEXT,
  created_at TIMESTAMP,
  expires_at TIMESTAMP,
  view_count BIGINT,
  PRIMARY KEY (user_id, story_id)
) WITH CLUSTERING ORDER BY (story_id DESC)
AND default_time_to_live = 86400;  // Auto-delete after 24h

CREATE TABLE story_views (
  story_id TIMEUUID,
  viewer_id BIGINT,
  viewed_at TIMESTAMP,
  PRIMARY KEY (story_id, viewer_id)
) WITH default_time_to_live = 86400;

CREATE TABLE user_stories (
  user_id BIGINT,
  story_id TIMEUUID,
  created_at TIMESTAMP,
  PRIMARY KEY (user_id, story_id)
) WITH CLUSTERING ORDER BY (story_id DESC);

TTL Strategy

Cassandra's native TTL (default_time_to_live) automatically deletes expired rows after 24 hours. No manual cleanup required. However, storage is not immediately reclaimed (tombstones linger for grace period). Enable date-tiered compaction to efficiently delete old data.

Story Feed Algorithm

  1. Get Followed Users: Query follows table for all users the viewer follows.
  2. Fetch Stories: For each followed user, query stories table (user_id, last 3 stories). Cassandra returns them in reverse chronological order (DESC clustering).
  3. Filter Active: Filter stories by created_at > now - 86400 seconds.
  4. Sort: Order by creation time (most recent first).
  5. Cache: Cache story list in Redis for 10 seconds (user's viewed stories frequently).

View Receipts (Who Viewed My Story)

When user views story:

  1. Record view in story_views table: (story_id, viewer_id, viewed_at).
  2. Increment view_count (denormalized in stories table). Use Cassandra counter or increment async.
  3. Push notification: "X viewed your story" (aggregated, e.g., "5 people viewed your story").

To fetch story views: 

SELECT viewer_id, viewed_at FROM story_views WHERE story_id = ? LIMIT 100

This returns viewers in insertion order (not by time viewed). To get ordered by time, add secondary index or store in separate table with time as clustering key (but requires denormalization).

Storage & Bandwidth

Stories are typically same size as photos (~2 MB). But TTL means:

30M stories/day (3× photos) × 2 MB = 60 TB/day written
Auto-expire after 24h, so total storage: 60 TB × 1 day = 60 TB max
Much smaller than permanent photos (~3.8 EB/year)

Tiered Storage (Cold Tier)

Stories older than 6 hours can be moved to cheaper cold storage (Glacier). Original is kept in S3 standard for fast delivery, but retrieved data is already 6 hours old so viewers likely moved on. If user re-views an old story (rare), fetch from Glacier (3-hour retrieval latency).

Search & Explore

Search Components

  • User Search: Find users by username, bio keywords. Served from MySQL (indexed on username).
  • Hashtag Search: Find posts by hashtag. Served from Elasticsearch (inverted index) or Redis (trending hashtags, Sorted Set).
  • Location Search: Find posts by location. Geospatial index in Elasticsearch.
  • Caption/Tag Search: Full-text search on photo captions and user tags.

Trending Hashtags

Maintained in Redis Sorted Set with expiry: 

// Every time a photo is tagged with #vacation, increment
ZINCRBY trending_hashtags 1 "vacation"
EXPIRE trending_hashtags 86400  // Reset daily

// Fetch top 10 trending
ZREVRANGE trending_hashtags 0 9 WITHSCORES

// Result: [["vacation", 100000], ["travel", 95000], ...]

Explore Page Personalization

The Explore page shows content not from followed users. Ranking:

  1. Interest Matching: ML model predicts user interests from past likes (tags, creators, locations).
  2. Trending Boost: Posts with hashtags in trending list get boost.
  3. Engagement Filter: Only show posts with >100 likes (quality filter, reduce spam).
  4. Diversity: Mix of hashtags, creators, locations to avoid repetition.
  5. Personalized Ranking: Same ML model as feed, but on broader candidate set (top 10k trending posts).

Explore content is cached per user in Redis for 1 hour (cheaper than ranking 10k posts every refresh).

Search Indexing Pipeline

When photo is posted:

  1. Kafka event published: { photo_id, user_id, caption, tags, location }.
  2. Indexer consumer: Parses hashtags from caption (#vacation → "vacation"), publishes to Elasticsearch.
  3. Index is near real-time (2-5 second delay).
  4. Indexing failure triggers dead-letter queue (DLQ) and alert.

Cache Invalidation Strategy

On Photo Like:

  1. Write like to Cassandra (primary).
  2. Increment photo:{photo_id}:like_count in Redis.
  3. When like_count changes significantly (+100 likes), invalidate trending_photos (refresh from Cassandra).

On Follow:

  1. Write to follows table in Cassandra.
  2. Invalidate follower's feed:{user_id} in Redis (expires in 100 sec anyway).
  3. Clear feed cache immediately if using <100 seconds TTL.

On Photo Delete:

  1. Mark as deleted in Cassandra (soft delete, update is_deleted flag).
  2. Delete from feed:{follower_id} for all followers (async job, may be queued).
  3. Delete from trending_photos, search index immediately.

Cache Warming

For celebrities, pre-warm feed cache:

  • Batch job runs every hour: for top 1M users (by follower count), populate feed:{user_id} with most recent posts from followed users.
  • Avoids cold cache miss on first feed refresh.
  • Estimated cost: 1M users × 1 KB feed = 1 TB RAM (acceptable given Redis cluster size for Instagram).

Handling Cache Failures

If Redis is down:

  • Circuit breaker: after 3 consecutive failures, disable cache lookups for 30 seconds.
  • Fall back to DB: query Cassandra directly (slower, but maintains availability).
  • Degrade gracefully: return fewer results (top 10 photos instead of 20) to reduce load on DB.
  • Alert ops when cache miss rate exceeds 10%.

Cache Efficiency Metrics

Monitor:

  • Hit Rate: (Cache hits) / (Cache hits + misses). Target 80%+.
  • Eviction Rate: Items evicted due to memory limit. Should be <5%.
  • Memory Usage: Total bytes in cache. Alert if >90% of allocated memory.
  • Latency: p99 cache lookup <5ms. p99 DB fallback <100ms.

Notifications

Notification Types

  • Likes: "3 people liked your photo."
  • Comments: "Sarah commented on your photo: 'Love it!'"
  • Follows: "John followed you."
  • Mentions: "@Sarah mentioned you in a comment."
  • Direct Messages: "Sarah sent you a message."

Notification Pipeline

  1. Event Generation: When photo is liked, post to Kafka topic "photo_liked" with event: { photo_id, user_id, liker_id, timestamp }.
  2. Aggregation Service (Kafka Consumer): Consumes photo_liked events. Groups by (photo_id, liker_id) and aggregates within 10-second window: 
    // Group likes on same photo
{
  photo_id: "abc123",
  like_count: 3,
  liker_ids: ["user_1", "user_2", "user_3"],
  timestamp: 2026-03-18T10:00:00Z
}
  3. Deduplication: Check if user already received notification for this photo in last 24 hours. If yes, update count instead of sending new notification. Redis set: { notification_id, 24h TTL }.
  4. Dispatch Service: Sends aggregated notification to APNs (iOS) and FCM (Android). Returns delivery status (delivered, bounced, invalid token).
  5. Notification Storage: Store in Cassandra for notification center (in-app view of past notifications). Retention: 30 days.

Notification Aggregation Example

Without aggregation: 100 people like a photo → 100 notifications → notification spam.

With aggregation (10-second window):

  • First like at t=0: Send "Sarah liked your photo."
  • Likes at t=1 to t=10: Aggregate, send at t=10: "Sarah, John, and 98 others liked your photo."
  • Likes after t=10: Send new notification: "And 5 more people liked your photo."

Notification Delivery Guarantees

  • At-least-once: Notification may be delivered multiple times if service crashes and replays Kafka. Mitigate with deduplication ID in APNs/FCM.
  • Ordering: Notifications are delivered in order per user (single consumer partition per user_id).
  • Latency: <5 second end-to-end from event to user receiving push.

Notification Preferences

Users can control:

  • Notification types (likes, comments, follows).
  • Quiet hours (e.g., no notifications 10 PM - 8 AM).
  • VIP list (always notify for follows from VIPs).

Preferences stored in MySQL user table:

  notify_likes BOOLEAN,
  notify_comments BOOLEAN,
  notify_follows BOOLEAN,

);

Metrics

  • Delivery Rate: % of notifications successfully delivered to APNs/FCM. Target >99%.
  • Bounce Rate: % of invalid tokens (user uninstalled app, opted out). Monitor for excessive bounces; invalidate tokens.
  • Latency: Time from event to user seeing notification. p99 <5 seconds.
  • Click-Through Rate: % of users who tap notification to open app. Indicates relevance.

Trade-offs & Design Decisions

Consistency vs. Availability

Data Type Consistency Choice Rationale Trade-off
User Auth Strong (MySQL + replication) Must not grant access to wrong user Lower availability if DB down; circuit breaker + cached tokens
Like Count Eventual (Cassandra counter) Approximate counts acceptable; precision not critical User may see off-by-few like counts for 1-2 minutes
Feed Timeline Eventual (Redis cache) Feed is not a legal document; stale by 1-2 minutes is OK Deleted posts may briefly appear; fans may miss recent posts
Follow Relationship Strong (Cassandra with TWO_QUORUM) Follow = relationship; must be accurate for privacy and feed correctness Slower writes (wait for 2 of 3 replicas); acceptable for infrequent updates

Push vs. Pull (Feed Fan-Out)

Aspect Pure Push Pure Pull Hybrid (Chosen)
Write Latency Slow (fan-out millions) Fast Medium (push <1M only)
Read Latency Very fast Slow Fast (Redis) + merge (DB)
Storage Huge (per-user feeds) Small Medium (tuned by threshold)
Spiky Load Celebrity posts spike to 10M writes/sec Celebrity posts spike reads, not writes Balanced: write spikes <2 minutes, read spikes managed by ranking

Photo Format & Compression

Choice: Store original photo + pre-generated thumbnails (3 formats).

  • Pro: One-time transcoding cost. Feed loads fast (thumbnail only). User sees original on demand.
  • Con: 3× storage cost. Transcoding delays (2-5 seconds).
  • Alternative: On-demand transcoding via CDN. Con: slow on first request, transcoding infra scaling challenge.

Database: Cassandra vs. DynamoDB

Aspect Cassandra (Chosen) DynamoDB
Cost (at Instagram scale) Lower (self-managed, commodity hardware) Higher (AWS managed premium)
Operational Overhead High (manage nodes, repairs, upgrades) Low (AWS manages)
Consistency Controls Fine-grained (tunable quorum) Limited (eventually consistent by default)
Global Secondary Indexes None (must denormalize) Native support
Best Case Well-understood ops, cost savings at massive scale Faster time-to-market, no ops burden

Kafka vs. RabbitMQ for Event Bus

  • Kafka (Chosen): High throughput (millions of events/sec), replayable (event sourcing), excellent for fan-out (multiple consumers).
  • RabbitMQ: Lower latency (ms vs. seconds), better for request-reply patterns, but lower throughput.
  • Decision: Instagram has massive event volume (100M photos/day) and needs replay for reprocessing. Kafka is better.

Interviewer Follow-Up Questions & Answers

Q: How do you handle duplicate uploads?

A: Use perceptual hashing (pHash) of the image. Before issuing presigned URL, check if user uploaded visually identical photo in last 24h. If match found, return existing photo_id instead of creating duplicate. Store hash in Redis with 24h TTL. Collision probability for Instagram scale: <0.01%.

Q: What if a photo gets deleted? How fast does it disappear from followers' feeds?

A: Soft-delete: mark is_deleted=true in Cassandra (don't physically delete yet, preserve for compliance). Immediately remove from Redis feed caches (invalidate feed:{follower_id}). Background job (runs every 5 minutes) scans all feed caches for deleted photos and removes them. Acceptable latency: photo invisible within 10 seconds to users who refresh feed; within 5 minutes for cached versions. This balances user experience (deletion feels fast) with operational simplicity (no need to track every deleted photo to every follower).

Q: How do you prevent spam/bot accounts from artificially inflating like counts?

A: Multiple layers:

  • Bot detection: ML classifier analyzes account behavior (age, login patterns, like/comment distribution). Flag suspicious accounts as likely bots.
  • Like filtering: Don't count likes from flagged bots in official like count. Store separately in analytics DB.
  • Rate-limiting: Max 100 likes/min per user. Violators throttled for 1 hour.
  • Anomaly detection: If photo likes spike 100× above creator's historical average in 1 minute, flag for manual review.

Q: What's the latency breakdown for a feed refresh?

A:

  • Network latency (client to LB): 50ms
  • API Gateway + Auth check (Redis): 5ms
  • Feed Service (Redis cache hit): 5ms
  • Fetch photo metadata from Cassandra (parallel, 20 photos): 10ms
  • Ranking (ML inference, TensorFlow Serving): 20ms
  • Network latency (server to client): 50ms
  • Total: ~140ms (p50). p99: ~400ms (includes Cassandra cache misses, GC pauses).

Q: How do you scale the notification system to handle 1B notifications/day?

A:

  • Aggregation: Group likes on same photo within 10-second window. Reduces 1B likes to ~200M aggregated notifications.
  • Batching: Send batch of notifications to APNs/FCM (not one-by-one). Reduce from millions of RPC to thousands.
  • Partitioning: Kafka topic partitioned by user_id. Each partition is consumed by separate worker. Horizontal scaling = add more workers.
  • Throughput: Each worker can dispatch ~10k notifications/sec (assume 100ms per APNs call, 10 concurrent connections). 100 workers = 1M notifications/sec → 1B/day covered.
  • Cost: ~100k for APNs/FCM charges per month (at Instagram scale).

Q: How do you handle the "thundering herd" problem when a celebrity posts?

A: Avoid it with the hybrid push/pull approach:

  • Celebrity post (>1M followers) goes straight to DB, NOT fanned out. No spike in writes.
  • When followers refresh feed, they query DB for celebrity's recent posts. Spread reads over time (not all at once).
  • Cache the merged feed result (regular + celebrity posts) for 10 seconds in Redis. Subsequent reads hit cache (no DB hit).
  • If DB read spike still occurs (10x normal traffic), scale read replicas horizontally and/or use read-only secondary DC.

Q: How do you ensure feed freshness for real-time notifications?

A:

  • Push-based for followed users: Post visible in Redis feed within 5 seconds of posting.
  • Pull-based for celebrities: Post visible in next feed refresh (user-initiated), typically <10 seconds.
  • SLA: Post visible to at least 95% of followers within 2 seconds. Measured by tracing post through fan-out pipeline.
  • WebSocket for live updates (optional): If user has app open, push notification of new posts to client in real-time (not just on feed refresh).

Q: What happens if Cassandra goes down for 1 hour?

A:

  • Circuit breaker (after 3 failures, stop trying for 30s): Prevent cascading failures.
  • Reads: Return stale data from Redis cache (feed is 1-2 hours old but still usable).
  • Writes: Queue to Kafka. Photo posts are eventually written to Cassandra when it recovers. <1 hour acceptable for eventual consistency.
  • Availability: Feed is degraded (stale) but still available. Upload fails after 3 retries; user sees error and can retry later.
  • Recovery: Cassandra comes back. Hinted handoff replays queued writes. Eventually consistent replication catches up in <5 minutes.

Q: How do you handle geographic distribution? (US, EU, Asia).

A:

  • Multi-region deployment: Independent API servers in US-EAST, EU-WEST, AP-SOUTHEAST.
  • Cassandra: Distributed across 3 DCs (US, EU, Asia) with RF=3. Reads from local DC (low latency). Writes to all 3 (quorum = 2 of 3).
  • MySQL: Primary in US-EAST. Read-only replicas in EU, Asia. Writes only to primary (then replicated).
  • Kafka: Cluster-to-cluster replication (MirrorMaker) between regions. Events published to all regions within 5 seconds.
  • CDN: Global CDN (CloudFront) with regional edge caches. Photo served from nearest edge (50ms latency from user).
