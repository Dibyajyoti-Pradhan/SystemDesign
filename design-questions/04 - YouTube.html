Design YouTube

A staff-engineer level system design for a global video streaming platform

📌Overview

YouTube is a massive-scale video hosting, streaming, and discovery platform serving billions of users globally. The system must handle:

  • Continuous video uploads from creators worldwide
  • Billions of concurrent viewers with sub-second latency
  • Petabytes of video content in multiple formats and bitrates
  • Real-time recommendations and search indexing
  • Complex engagement signals (comments, likes, subscriptions)

Scale snapshot: 2 billion users, 500 hours/min uploaded, 1 billion hours watched daily.

Key Insight: YouTube's complexity lies not in a single novel algorithm, but in orchestrating hundreds of services that handle upload, transcode, stream, cache, search, and recommend content at planetary scale with sub-second latency.

📌Clarifying Questions

Question Answer
Live streaming? Yes. Assume 10% of concurrent viewers watch live. Require <50s latency for live feeds.
DRM/Copyright? Assume DRM via Widevine for premium; focus architecture on encryption pipeline. Copyright detection is a separate ML service.
Offline download? Yes, in mobile app. Downloaded videos expire after 30 days.
Regional requirements? Design for global CDN; some regions block/throttle (e.g., China). Use Cloudflare, Akamai, or Open Connect.
Subtitles/Closed captions? Auto-generate via ML; allow user uploads. Treat as separate asset with versioning.
Monetization? Ad placement service is separate; focus on content delivery. Track watch time for creator payouts.
Analytics? Real-time viewer counts, audience retention, thumbnails A/B tests. Metrics pushed to creators in real-time via WebSockets.

📌Functional Requirements

Core Features

  • Video Upload: Support up to 12 GB per file, resumable, progress tracking.
  • Video Streaming: Multi-bitrate (144p–4K), adaptive bitrate selection, fast start (<2s).
  • Search: Full-text search on title, description, transcripts; autocomplete; trending.
  • Recommendations: Homepage feed, "Up Next", sidebar recommendations; personalized.
  • Engagement: Like, comment, share, subscribe. Nested comments with moderation.
  • Live Streaming: Real-time broadcast with chat, super chat (monetization).
  • Offline Download: Mobile app download for 30-day offline view.
  • Playback History: Store and retrieve user watch history; "Continue Watching".

Edge Cases & Constraints

  • Viral videos: Sudden 1000x traffic spike; rely on CDN push and cache warming.
  • Regional blackouts: Content region-locked; enforce geo-blocking in CDN.
  • Copyright strikes: Disable monetization, demonetize, or take down video.
  • Spam comments: ML-based filtering; report and remove.
  • Long uploads: Videos >24 hours; resumable upload essential.
  • Low-bandwidth regions: Adaptive bitrate must drop to 144p for <500 Kbps.

📌Capacity Estimation

User & Traffic

  • DAU: 500M daily active users
  • Concurrent viewers peak: 50M simultaneous
  • Uploads: 500 hours/minute globally
  • Watch time: 1 billion hours/day

Raw Storage Calculation

Uploaded per day = 500 hours/min × 60 × 24 = 720,000 hours/day

Raw file size (10 Mbps average bitrate for source):
  720,000 hours × 3,600 sec/hour × 10 Mbps / 8 = 32.4 PB/day raw storage

At 365 days/year × 10 years retention = 118,260 PB = 118 EB

Transcoded Output

Resolutions: 144p, 240p, 360p, 480p, 720p, 1080p, 4K
Codec: H.264 (bitrate ≈ resolution scaling factor)

144p: 0.5 Mbps  |  240p: 1 Mbps  |  360p: 2.5 Mbps  |  480p: 4 Mbps
720p: 8 Mbps    |  1080p: 15 Mbps | 4K: 40 Mbps

Total bitrate across all versions ≈ 70 Mbps
Transcoded storage per day = 720k hours × 3600 × 70 Mbps / 8 = 225 PB/day
10-year retention = 822,750 PB ≈ 823 EB

Streaming Bandwidth Peak

1 billion hours watched/day ÷ 86,400 sec = 11.6M hours streaming per second
Average bitrate = 4 Mbps (mix of 360p-1080p)
Bandwidth = 11.6M × 4 Mbps = 46.4 Tbps egress

With CDN cache hit rate 95%+, origin carries ~2.3 Tbps
Cost at $0.05/GB ≈ $14.4M/month egress alone

Database Size

Video metadata (title, desc, creator, length, etc):
  500M videos × 2 KB = 1 TB (MySQL)

Comments:
  5 billion comments × 500 bytes = 2.5 TB (Cassandra)

User engagement (likes, views):
  50B view events × 100 bytes = 5 TB (Cassandra)
Cost Reality: Storage ($10–30 per TB/month) and egress bandwidth (largest cost) dominate infrastructure spend. Optimizing codec choice and cache hit rate is critical.

📌High-Level System Architecture

Tier Explanation

  • Client: Web browsers, mobile apps, smart TVs. Use WebSockets for live chat.
  • CDN (Edge): Global edge locations (Akamai, Cloudflare, AWS CloudFront). Cache video segments and metadata.
  • API Gateway: Rate limiting, auth (OAuth2), request routing. Horizontal scaling.
  • Core Services: Microservices for upload, transcoding orchestration, streaming, search, recommendations, comments, notifications.
  • Storage: S3 for raw/transcoded videos, MySQL for metadata, Cassandra for high-write engagement data (comments, likes), Redis for cache/counters, Elasticsearch for full-text search, Kafka for event streaming.
  • Async Queue: Kafka manages background jobs (transcoding, notifications, analytics).
Key Design Point: The 95%+ CDN cache hit rate is what makes YouTube economically viable. Origin serves only cache misses; the long tail of videos must be pushed to edge on first upload.

📌Video Upload & Ingestion

Upload Protocol: TUS (Resumable Upload)

POST /upload
Content-Type: application/offset+octet-stream
Upload-Length: 5368709120  # 5 GB
Tus-Resumable: 1.0.0

Response: 201 Created
Location: /upload/abc123def456
Upload-Offset: 0

Chunking Strategy

  • Chunk size: 5 MB per chunk. Balances resume granularity vs. overhead.
  • Retry logic: Exponential backoff. Max 3 retries per chunk.
  • Resumable: Client tracks offset; server validates. On reconnect, resume from last offset.
  • Deduplication: Compute perceptual hash (phash) on raw file; detect duplicate uploads. Share transcoded output if hash matches.

Upload Flow

  1. Creator opens upload form; gets pre-signed S3 URL (valid 24h).
  2. Client chunks video into 5 MB pieces, uploads to S3 with resumable protocol.
  3. S3 fires event (SNS/SQS) when upload completes.
  4. Upload Service writes metadata to MySQL (video_id, creator_id, title, raw_s3_path).
  5. Publishes job to Kafka: `{video_id, raw_s3_path, resolutions=[144p, 240p, 360p, 480p, 720p, 1080p]}`
  6. Transcoding Orchestrator picks up job, spawns transcoding tasks.
Note: TUS protocol is industry-standard for resumable uploads. Implement server-side state tracking: `{upload_id → {offset, filename, hash}}` in Redis. Expires after 7 days inactivity.

📌Transcoding Deep Dive

Why Transcode?

  • Multi-device delivery: A phone needs 360p; a 4K TV needs 4K. Encode once, store all versions.
  • Adaptive streaming: Player switches bitrate based on bandwidth (ABR). Can't do this with single source.
  • Quality control: Normalize audio levels, fix color, apply compression algorithms per codec.
  • Format conversion: User uploads QuickTime; we deliver HLS/DASH in H.264.

Codec Comparison

Codec Encoding Time (1 GB) Bitrate @ 1080p Device Support Use Case
H.264 45 min (baseline) 8–12 Mbps 100% (universal) Main production codec. Safe default.
VP9 120 min (2.7x slower) 6–8 Mbps (20% better) ~85% (Chrome, Firefox) High-efficiency; skip less powerful devices.
AV1 300+ min (6x+ slower) 4–6 Mbps (40% better) ~40% (emerging) Future codec; batch offline use (YouTube premieres).
HEVC/H.265 90 min 6–8 Mbps ~70% (licensing cost) Mobile optimized; Apple devices. Patent risk.

Resolution Ladder

144p (256×144):    0.5 Mbps   |  240p (426×240):   1 Mbps
360p (640×360):    2.5 Mbps  |  480p (854×480):   4 Mbps
720p (1280×720):   8 Mbps    |  1080p (1920×1080): 15 Mbps
1440p (2560×1440): 25 Mbps   |  4K (3840×2160):   40 Mbps

DAG-Based Transcoding Pipeline

Input Video
    ├─ Extract audio → Audio encode (AAC 128 Kbps)
    ├─ Video stream
    │  ├─ Resize to 144p → H.264 encode → 144p.mp4
    │  ├─ Resize to 360p → H.264 encode → 360p.mp4
    │  ├─ Resize to 720p → H.264 encode → 720p.mp4
    │  └─ Resize to 1080p → H.264 encode → 1080p.mp4
    └─ Frame extraction (1 per 5s) → Thumbnail resize → thumb.jpg

Tasks can run in parallel; frame extraction blocks on video decode.

Transcoding Infrastructure

  • Orchestrator: Kubernetes or Airflow DAG. Submits tasks to worker pool.
  • Workers: ffmpeg/libx264 on GPU-accelerated instances. 1 worker ≈ 10 concurrent transcode jobs.
  • Cost per video: 1 GB video × 7 resolutions × $0.05/compute hour ≈ $0.35–0.50 per video.
  • Parallelism: Each resolution in separate task. Audio encoded once, muxed into each video version.
  • Priority queue: Monetized/viral videos prioritized. Backlog in Redis.

Failure Handling

  • Idempotent jobs: Use video_id + resolution as task key. If worker crashes mid-transcode, restart; write to S3 atomically (multipart upload).
  • Timeout: 2 hours max per task. If exceeded, retry on different worker or escalate.
  • Fallback: If transcoding fails 3 times, notify creator. Default to H.264 1080p only if all fail.
Key Optimization: Batch transcode jobs during off-peak hours (2 AM–6 AM UTC). Reduces cost 30–40% via spot instances. For monetized videos, prioritize immediate turnaround.

Player Buffer Algorithm

  1. Measure available bandwidth (exponential moving average over last 10 segments).
  2. Select bitrate: choose highest bitrate ≤ available bandwidth with safety margin (70% of BW).
  3. Prefetch segments into buffer (2–5 segments ahead).
  4. Monitor buffer level:
    • If buffer < 2 segments: rebuffer (pause playback, refill).
    • If buffer > 5 segments: can afford bitrate upgrade on next segment.
  5. Stall avoidance: always fetch next segment preemptively.

Key Metrics

  • Time to First Frame (TTFF): <2 seconds (p99). Goal: initial buffer of 3 segments.
  • Stall rate: <0.5% (fewer than 1 stall per 200 plays). Each stall ≈ 5% watch time loss.
  • Bitrate switch frequency: Avoid > 3 switches per minute (jarring to user).
  • Avg bitrate: Aim for 80% of available BW (good QoE without excessive rebuffering).
Pro Tip: Implement "smart prefetch": if video is viral and CDN origin is congested, switch to lower bitrate preemptively to avoid stalls. Metrics > video quality trade-off.

📌CDN Architecture

CDN Workflow: Origin Pull vs. Push

Aspect Origin Pull (On-Demand) Push Preload
Trigger User requests segment; CDN misses and fetches from origin. Transcode completes; system proactively pushes to CDN edge.
Latency First user sees origin fetch latency (~200–500 ms). First user hits cache (10–50 ms). Much faster.
Cost Only pay for actual traffic. Efficient for long tail. Pay to push unused content. Waste for unpopular videos.
Use case Niche, old videos. 80% of videos get <1k views. Viral, new uploads, trending. Fast initial experience critical.

Hybrid Strategy

  • New upload: Push high-bitrate (1080p) to top-100 CDN PoPs globally.
  • If video trends (1M views in 24h): Push all resolutions to all PoPs.
  • If video unpopular (<10k views in 7 days): Pull-only from origin; CDN auto-purges.
  • Cache TTL: 30 days for actively viewed, 7 days for cold videos.

Multi-Tier Cache Hierarchy

User Request
    ↓
Edge PoP Cache (50 ms latency, max 5 PB)
    ↓ [MISS]
Regional Cache (150 ms latency, max 20 PB)
    ↓ [MISS]
Origin S3 (500 ms latency, unlimited but expensive)

Open Connect (YouTube's ISP Peering)

  • Concept: YouTube installs mini data centers (8–20 TB appliances) directly in ISP networks.
  • Advantage: Eliminates ISP-to-backbone link congestion. Local cache hits = <10 ms latency, ~$0 CDN cost.
  • Deployment: Tier-1 ISPs get hardware. During off-peak (2 AM–6 AM), sync hot content via overnight prefetch.
  • Fallback: If Open Connect cache fails, fall through to regional CDN.

Cache Warming for Viral Videos

  1. Detect viral signature: views/hour > threshold (e.g., 10k views/hour).
  2. Publish event to Kafka: `{video_id, bitrates_to_warm=[360p, 720p, 1080p]}`
  3. Trigger CDN push job: fetch all segments from S3, distribute to top-200 PoPs.
  4. Parallel push (not sequential) via batch API. ~30 min to warm globally.
  5. Monitor cache hit rate; if <95% after warm, trigger 4K push as well.

Long-Tail Optimization

  • 80% of views: 20% of videos (power law distribution).
  • Strategy: Only push top-1% trending videos. Rest use origin pull with regional cache tier.
  • Cost savings: Reduces CDN push cost by 60%, saves $2B/year at YouTube scale.
  • Trade-off: First user of niche video sees higher latency (~200 ms), but acceptable.
Reality Check: CDN egress is your largest cost (~$14M/month at YouTube scale). Every 1% cache miss increase costs ~$150k/month. Cache hit rate obsession is justified.

📌Database Design

Videos Table (MySQL, Sharded by video_id)

CREATE TABLE videos (
    video_id BIGINT PRIMARY KEY,  -- UUID-like, globally unique
    creator_id BIGINT NOT NULL,   -- Foreign key
    title VARCHAR(255) NOT NULL,
    description TEXT,
    duration INT,                  -- seconds
    upload_timestamp DATETIME,
    thumbnail_url VARCHAR(512),
    view_count BIGINT DEFAULT 0,   -- cached, async sync from Cassandra
    like_count BIGINT DEFAULT 0,
    comment_count BIGINT DEFAULT 0,
    status ENUM('uploading', 'transcoding', 'ready', 'deleted'),
    raw_s3_path VARCHAR(512),      -- e.g., s3://youtube-raw/abc123def...
    transcoded_s3_path VARCHAR(512),
    is_monetized BOOLEAN DEFAULT FALSE,
    is_private BOOLEAN DEFAULT FALSE,
    dislikes_disabled BOOLEAN DEFAULT TRUE,  -- post-2021 change
    created_at DATETIME DEFAULT NOW(),
    updated_at DATETIME DEFAULT NOW(),

    INDEX idx_creator_id (creator_id),
    INDEX idx_status (status),
    INDEX idx_created_at (created_at)
);

Sharding Strategy

  • Shard key: video_id % NUM_SHARDS (e.g., 256 shards).
  • Rationale: Distributes writes/reads evenly. Queries by creator_id hit secondary index (slower but acceptable).
  • Growth: Start with 16 shards, add shards online as data grows (range-based migration).

Users Table (MySQL, Global, No Sharding)

CREATE TABLE users (
    user_id BIGINT PRIMARY KEY,
    username VARCHAR(30) UNIQUE NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    profile_image_url VARCHAR(512),
    subscriber_count BIGINT DEFAULT 0,  -- cached
    created_at DATETIME DEFAULT NOW(),
    INDEX idx_email (email)
);

Comments Table (Cassandra, High Write Volume)

CREATE TABLE comments (
    video_id BIGINT,           -- partition key
    comment_id BIGINT,         -- clustering key (timestamp-based)
    author_id BIGINT,
    parent_comment_id BIGINT,  -- for replies, NULL for top-level
    text TEXT,
    like_count BIGINT,         -- approximate
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    is_deleted BOOLEAN DEFAULT FALSE,
    PRIMARY KEY (video_id, comment_id)
) WITH CLUSTERING ORDER BY (comment_id DESC);

Likes/Engagement (Redis Counter + Cassandra Flush)

  • Real-time: Increment Redis counter (atomic). Sub-millisecond response.
  • Async flush: Every 5 min, batch flush Redis counters to Cassandra (eventual consistency).
  • Recovery: On Redis failure, rebuild from Cassandra + last flush timestamp.
  • Key pattern: `like_count:{video_id}` → counter value.

Subscriptions (Cassandra)

CREATE TABLE subscriptions (
    subscriber_id BIGINT,      -- partition key
    creator_id BIGINT,         -- clustering key
    subscribed_at TIMESTAMP,
    PRIMARY KEY (subscriber_id, creator_id)
);

-- Reverse index for notifications
CREATE TABLE creator_subscribers (
    creator_id BIGINT,         -- partition key
    subscriber_id BIGINT,
    subscribed_at TIMESTAMP,
    PRIMARY KEY (creator_id, subscriber_id)
);

Rationale Summary

Table Store Why
Videos MySQL (sharded) Structured, moderate write, lots of reads. ACID guarantees for video status transitions.
Users MySQL (global) Small dataset (<10 GB). Auth queries latency-critical. Replication to replicas sufficient.
Comments Cassandra Ultra-high write (5B events/day), only time-range queries. Denormalization OK.
Engagement (likes/views) Redis (hot) + Cassandra (cold) Extreme read/write velocity. Redis for instant response. Cassandra for durable counters.
Subscriptions Cassandra Time-series, range queries by creator. Fanout notifications require reverse index.
Key Insight: YouTube's data model is not a single database. Each data type lives in the storage optimized for its access pattern: relational for metadata, column-store for time-series, cache for hot counters.

📌Video Search

Elasticsearch Indexing

PUT /videos/_mapping
{
  "properties": {
    "video_id": { "type": "keyword" },
    "title": { "type": "text", "analyzer": "standard" },
    "description": { "type": "text", "analyzer": "standard" },
    "creator_name": { "type": "text" },
    "tags": { "type": "keyword" },
    "upload_date": { "type": "date" },
    "view_count": { "type": "long" },
    "like_count": { "type": "long" },
    "engagement_score": { "type": "float" },
    "transcript": { "type": "text", "analyzer": "standard" },
    "thumbnail_url": { "type": "keyword" }
  }
}

Ranking Algorithm: TF-IDF + Engagement

score = TF-IDF(query, document) × (1 + engagement_factor)

engagement_factor = (view_count ^ 0.5 + like_count ^ 0.7) / (days_old + 1)

Example: "machine learning tutorial"
- TF-IDF component: matches on title/transcript
- Engagement boost: 10M views → 100x boost. 1M views → 32x boost.
- Recency: video from 1 week ago vs. 1 year ago (1-year loses factor of ~52)

Autocomplete (Prefix Search)

  • Data source: Trie + Redis. Top 10k searches cached.
  • Request: User types "machine"; return top suggestions in <100 ms.
  • Prefix query (Elasticsearch): 
    GET /autocomplete/_search
{
  "query": {
    "match_phrase_prefix": {
      "query_text": "mach"
    }
  },
  "size": 10
}
  • Real-time refresh: Log all queries to Kafka. Batch update autocomplete trie every hour (with freshness vs. stale trade-off).

Trending Algorithm

  1. Capture view velocity per video (views in last 24h / views in total).
  2. Apply smoothing: velocity = (views_24h ^ 0.8) / (days_since_upload + 1)
  3. Bias toward verified creators to avoid spam gaming.
  4. Generate "Trending" playlist: top 50 videos by velocity, refresh every 6 hours.
  5. Regional variation: trending in US ≠ trending in India (different creator base, content culture).

Search Latency Budget

Total search time: P99 < 200 ms

Elasticsearch query: 80 ms (p99)
  - Parse query: 1 ms
  - Inverted index lookup: 15 ms
  - Relevance scoring: 60 ms
  - Fetch & serialize: 4 ms

Cache + overhead: 120 ms
  - Redis prefix check: 10 ms
  - HTTP round-trip: 50 ms
  - Render: 60 ms

Scaling Search

  • Sharding: Partition videos index by video_id % NUM_SHARDS (8–16 shards).
  • Replication: 2–3 replicas per shard for read HA.
  • Query routing: Multi-shard query (broadcast-gather), merge results, re-rank on aggregator.
  • Caching: Cache top 100k queries + results in Redis. 30% of searches are repeats.
Pro Tip: For very large result sets (1M+ results), paginate with cursor-based iteration (keyset pagination), not offset. Offset requires computing all prior results; cursor just resumes from position.

📌Recommendation Engine

Two-Tower Architecture

User Embedding Tower              Video Embedding Tower
    ↓                                    ↓
user_id ────→ [embedding layer]        video_id ────→ [embedding layer]
watch_history → [LSTM]                 title ────→ [text encoder]
subscriptions → [attention]            duration ──→ [numeric features]
demographics ──→ [dense]               tags ─────→ [categorical]
                    ↓                                    ↓
                  [256-d vector]                     [256-d vector]
                    ↓                                    ↓
               Inner product (dot) ────────────────→ similarity score

Candidate Generation (Fast, Recall-Focused)

  • Method: Collaborative filtering via ANN (Approximate Nearest Neighbor) search.
  • Algorithm: Spotify's ANNOY or Faiss (Facebook). Index user embeddings, retrieve top 1k video candidates in <10 ms.
  • Features: Watch history (last 50 videos watched), subscriptions (videos from subscribed creators).
  • Update frequency: User embedding updated in real-time (every watch event), video embedding updated daily via batch.

Ranking (Precise, Slow, 1k → 20)

  • Method: LambdaMART or gradient boosted trees (XGBoost).
  • Features:
    • Two-tower similarity score
    • Video freshness (1 hour → boost, 1 month → decay)
    • Creator authority (verified badge, historical CTR)
    • Engagement signals (likes/comment ratio)
    • Diversity penalty (avoid redundant recommendations)
    • Ad-hoc signals (business rules, A/B test winner)
  • Latency budget: 50 ms (features are pre-computed, cached).

Real-time vs. Batch

Signal Freshness Requirement Approach
User watch event <5 sec (within session) Streaming (Kafka) → real-time user embedding update in Redis.
Video popularity (views/hr) <1 min Stream aggregation (Spark Streaming), update to feature store.
User embedding (model inference) <1 min Real-time service (KServe/TensorFlow Serving). Input: user_id → output: embedding vector. Cache in Redis.
Video embedding <24 hr (slower) Nightly batch job. Run on all 1B videos. Push updated embeddings to Faiss index.
Ranking model <1 hr (AB-test winners) Online learning (mini-batch updates), or A/B test daily winner becomes default.

Cold Start Problem

  • New user: No watch history. Fallback to geographic top charts + signup survey (preferred genres).
  • New video: No engagement yet. Use creator's historical content CTR + tags to bootstrap embedding.
  • New creator: Few videos, low subscriber count. Bootstrap from content tags + duration (cold start user likely interested in similar creators).

Diversity & Serendipity

  • Problem: Pure similarity can lead to echo chamber (all music videos → all recommendations are music).
  • Solution:
    • Re-rank top 100 to include 10–15% "diverse" picks (different category).
    • Use "exploration-exploitation" trade-off: 80% high-similarity, 20% random.
    • A/B test: "serendipity" arm gets more unrelated recommendations. Measure long-term retention.
Recommendation Paradox: YouTube maximizes watch time (metric), which biases toward addictive content over diverse learning. Ethical systems need explicit diversity constraints. Staff-level interviews often ask about this tension.

📌Comments & Engagement

Nested Comments Architecture

Video
├─ Top-level comment 1 (1k likes)
│  ├─ Reply 1 (100 likes)
│  ├─ Reply 2 (50 likes)
│  └─ [Load more replies...]
├─ Top-level comment 2
└─ [Load more comments...]

Data Model (Cassandra)

CREATE TABLE comments (
    video_id BIGINT,
    comment_id BIGINT,    -- sortable unique ID (snowflake / timestamp-based)
    author_id BIGINT,
    parent_comment_id BIGINT,  -- NULL for top-level, else = comment_id of parent
    text TEXT,
    like_count BIGINT,
    reply_count BIGINT,
    created_at TIMESTAMP,
    is_deleted BOOLEAN,
    PRIMARY KEY (video_id, comment_id)
) WITH CLUSTERING ORDER BY (comment_id DESC);

Sorting & Pagination

  • Top-level: Sort by "Relevance" (default) = like_count / (hours_old + 1), then by reply_count.
  • Replies: Sort by time (newest first) or likes (most liked first).
  • Pagination: Load 20 comments per page. Use cursor-based pagination (comment_id as cursor).
  • Expand replies: Load top 3 replies initially; "Load more replies" fetches next batch.

Like Counter Approximation

Real-time: Redis counter
    comment_like:{comment_id} ← atomic increment

Async flush (every 5 min):
    Batch read all counters → Cassandra UPDATE

Display logic:
    like_count = Redis_get(comment_like:{comment_id})
    if Redis_miss:
        like_count = Cassandra_select(comment_id).like_count
        Redis_set_with_ttl(comment_like:{comment_id}, like_count, 5min)
    return like_count

Spam & Moderation

  • Auto-filter: Regex patterns (phone numbers, spam URLs). Hash-based dedup (similar comments).
  • ML model: Binary classifier (spam vs. ham). Trained on historical spam reports.
  • User reports: Comments can be reported. Flagged comments hidden unless reported by multiple users / creator reviews.
  • Creator moderation: Creators can approve comments before display (if enabled). Batch approval via UI.
  • Shadow ban: Spam comments still saved but only author sees them (prevent user awareness).

Notification on Reply

  • Trigger: New reply published on user's top-level comment.
  • Event: Publish to Kafka: `{event: comment_reply, commenter_id, replier_id, video_id}`
  • Fanout: Notification Service consumes event. Sends push notification (mobile), in-app notification, email digest (if opted in).
  • Rate limit: Max 1 notification per commenter per day (avoid spam).
Moderation at Scale: 500M comments/day. Even 0.1% spam = 500k spam comments to review. Automated + human-in-the-loop hybrid is necessary. Pure automation misses context; pure humans can't scale.

📌Caching Strategy

Multi-Layer Cache

  • Layer 1 (Browser): HTTP caching headers. Thumbnails, manifests cached 1 hour locally.
  • Layer 2 (CDN Edge): Video segments, manifest files. TTL = 30 days. Cache key = video_id + resolution.
  • Layer 3 (Redis): Hot metadata (video_id → {title, creator, thumbnail_url}). TTL = 1 hour.
  • Layer 4 (In-app memory): Search results, recommendation rankings. Cleared on refresh.

What to Cache

Data Freshness TTL Layer
Video segments (m4s) Immutable (once transcoded) 30 days CDN edge
Manifest (m3u8) 5 min (static for published videos) 5 min CDN + Redis
Video metadata 5 min (title rarely changes) 1 hour Redis
Like count (display) 5 min 5 min Redis
Recommendations 1 min (user-specific) 1 min Redis (user_id → recommendations)
Search results 30 min (index updated async) 30 min Redis

Cache Invalidation Strategy

  • Time-based (TTL): Default for all caches. Simplest; eventual consistency.
  • Event-based:
    • User updates video title → publish event to Kafka → cache service deletes Redis key.
    • Creator uploads new video → invalidate homepage recommendations (depends on creator).
  • Manual purge: Creator can "hard refresh" cache (rare use case, for urgent corrections).
  • Cache stampede prevention: If key expires during high traffic, only one request recomputes; others wait (locking). Use probabilistic early expiration (recompute before TTL if accessed frequently).

Cache Key Design

-- Video metadata

youtube:video:{video_id}:like_count

-- User-specific
youtube:user:{user_id}:recommendations

-- Search
youtube:search:{query_hash}:results

-- Manifest (CDN)
youtube:manifest:{video_id}:{resolution}

Cache Hit Rate Goals

  • CDN segment cache: >95% (goal). Invest in aggressive push warming.
  • Redis metadata: >80% (acceptable). Metadata relatively static.
  • Search cache: >30% (lower; diverse queries). Top 100k searches hit 80% of traffic.
Key Insight: Cache hit rate is directly proportional to infrastructure cost. 5% improvement in CDN cache hit rate saves $50M/year at YouTube scale. Obsess over it.

📌Scalability & Failure Modes

Database Sharding & Growth

  • Videos table: 256 shards. Each shard ~4 PB (1B videos × 4 KB metadata). Replicate 3x → 12 PB per shard.
  • Resharding: When shard > 10 PB, split into 2 shards. Use consistent hashing range-based approach. Takes ~4 hours per shard pair.
  • Read replicas: 2 replicas per shard (1 primary + 2 secondary). Route reads to replicas; writes to primary (with replication lag ~100 ms).
  • Cross-DC: Replicate to 2nd data center for disaster recovery. Async replication; RTO ~1 hour.

Transcoding Worker Failure Recovery

Failure scenario: Worker crashes mid-transcode

Detection:
  Job heartbeat (worker pings every 30 sec)
  No heartbeat for 60 sec → mark job as FAILED

Recovery:
  Job re-queued to Kafka with retry_count += 1
  Assign to different worker
  Worker checks output S3 (idempotent): if partial file exists, resume from last checkpoint

Exponential backoff: 2^retry_count seconds before retry (max 4 retries)
After 4 failures: escalate to admin, notify creator "This video cannot be transcoded"

CDN Failover

  • Scenario: Primary CDN (Akamai) loses connection to origin S3.
  • Detection: CDN detects 503 from origin, fails over to secondary CDN (CloudFront) within 10 sec.
  • Traffic routing: DNS (Route53) health checks. If primary CDN fails health check, promote secondary.
  • Cache coherence: Secondary CDN caches independently. Slight duplication but ensures availability.

Database Read Replica Lag

  • Replication lag: ~100 ms typical, <1 sec p99.
  • Impact: User uploads video → metadata written to primary → reads from replica may see "not found" for 100 ms.
  • Mitigation:
    • Read-after-write consistency: for write operations, read from primary for 1 minute, then fall back to replica.
    • Or: cache written value in client-side session storage.

Cascading Failures: Thundering Herd

Scenario: Viral video suddenly gets 1M concurrent users. CDN cache miss → all hit origin.

Prevention:
  1. Cache warming: detect viral (see Viral Detection above), pre-push to CDN.
  2. Origin rate limiting: limit requests/sec from CDN to S3 (graceful degrade).
  3. Circuit breaker: if origin latency > 1 sec, CDN serves stale cache (cache-stale-on-error).
  4. Client-side backoff: if player gets 503, exponential backoff + alert user.

Fault Tolerance Hierarchy: Streaming (critical) > metadata (high) > transcode (medium) > search (low). Allocate engineering effort accordingly. A 1-hour transcoding delay is acceptable; a 1-second streaming latency spike is not.

Common Follow-Up Questions & Answers

Q: How do you handle live streaming?

A: Live requires <50 ms latency (vs. 100+ ms for VOD). Use protocol like RTMP (ingest) → transcode on-the-fly → HLS segments (2–3 sec segments, shorter window). Sliding window manifest (last 6 segments only). Lower quality (max 1080p) due to transcoding speed. WebSocket for live chat. Scale similar to streaming but with real-time constraint.

Q: What about copyright detection (e.g., music in video)?

A: Separate ML service (Content ID). During transcode, extract audio → fingerprint matching (Shazam-like). Compare against content database. If match: flag for review → monetize (if licensed) or demonetize (if not). Async job; happens in parallel with video encoding.

Q: How do you handle offline download in the mobile app?

A: Download service fetches video from CDN, stores encrypted on device. Encryption key is device-bound (stored in secure enclave). Video expires after 30 days (send notification before expiry). If user uninstalls app, offline content is lost. No offline DRM (Widevine license check requires network). Simpler than streaming DRM.

Q: How do you scale recommendations?

A: Two-tower model: candidate generation (ANN search, ~100 ms) → ranking (LambdaMART, ~50 ms). Batch compute video embeddings nightly. Real-time update user embeddings on watch event. Cache recommendations in Redis (user_id → top 100 videos) for 1 min. Fallback to geographic top charts if cache miss. A/B test new ranking models.

Q: What if creators upload duplicate content (e.g., uploading same movie twice)?

A: Perceptual hash (phash) on raw file. Detect duplicates before transcode. Can either: (1) reject upload ("video already exists"), (2) share transcoded output (save transcode cost), or (3) allow duplicates for derivative content (covers, reactions). YouTube mostly rejects or combines views under original.

Q: How do you prevent hotlinking (external sites embedding YouTube videos)?

A: Referer header check. Manifest served only to requests from youtube.com domain (and partner domains). External sites get 403. Embedded videos use iFrame with origin policy. Some partners (WordPress, Reddit) get whitelisted referers.

Q: What's your backup & disaster recovery strategy?

A: RTO (Recovery Time Objective): 1 hour. RPO (Recovery Point Objective): 5 minutes.

  • Database: 3x replication + cross-DC async replication. If primary DC fails, promote secondary (takes ~30 min).
  • S3: multi-region replication (S3 cross-region replication). Automatic failover to 2nd region.
  • Kafka: 3 replicas per topic. If leader fails, auto-elect new leader. No data loss.
  • Backup: weekly snapshots of MySQL to S3 Glacier (long-term cold storage). Can restore point-in-time within 24 hours.

Interview Tips: Show you understand YouTube is a **systems orchestration problem**, not a single algorithmic breakthrough. Discuss trade-offs (cost vs. latency, consistency vs. availability). Interviewers want to hear how you'd debug a production issue, not just textbook answers.
